# 核心数据模型

## 1. 模块概览

本文说明项目的核心持久化实体、运行时模型和它们之间的关系。数据分为两类：MySQL 持久化的 JPA 实体和仅在内存/会话中存在的运行时模型。

当前实现里需要特别注意五点：

- 用户和会话是核心归属关系：所有业务数据按 `userId` 隔离，会话属于用户，但沙箱资源按用户复用。
- `ChatMessage` 是运行时消息模型，`ChatMessageEntity` 是持久化实体，两者字段不完全对应。
- `UserSandboxEntity` 以 `userId` 为键（不是自增 ID），一个用户只有一条记录。
- TodoState 是会话内内存状态，不持久化到数据库。
- 向量数据存在 Milvus（可选），不在 MySQL。

## 2. 用户与会话

### 2.1 UserEntity

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | `Long` | 主键，自增 |
| `username` | `String(64)` | 用户名，唯一索引 |
| `passwordHash` | `String(128)` | BCrypt 密码哈希 |
| `token` | `String(64)` | 访问令牌，唯一索引 |
| `createdAt` | `LocalDateTime` | 创建时间 |

表名 `users`，索引 `idx_users_token` 和 `idx_users_username`（唯一）。

### 2.2 ConversationSessionEntity

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | `String(36)` | 会话 ID，UUID，主键 |
| `sandboxId` | `String(128)` | 当前会话绑定的沙箱 ID（不再作为资源归属来源） |
| `title` | `String(120)` | 会话标题，默认"新对话" |
| `userId` | `Long` | 会话所属用户 ID |
| `appId` | `Long` | 会话关联的 Agent 应用 ID |
| `enabledSkillIds` | `Set<String>` | 启用的技能 ID 集合 |
| `createdAt` | `LocalDateTime` | 创建时间 |
| `updatedAt` | `LocalDateTime` | 更新时间 |

表名 `conversation_session`。`sandboxId` 字段已弱化，AIO endpoint 统一由 `UserSandboxEntity` 管理。

### 2.3 ChatMessage 与 ChatMessageEntity

`ChatMessage` 是运行时消息模型（不可变，构造器传入），`ChatMessageEntity` 是持久化实体。

`ChatMessage` 关键字段：

| 字段 | 说明 |
| --- | --- |
| `role` | 消息角色（user/assistant/tool） |
| `content` | 消息内容 |
| `reasoning` | 思考链内容（reasoning_content） |
| `timestamp` | 时间戳（毫秒） |
| `files` | 文件附件列表（agent 回复时附带） |
| `toolCallId` | 工具调用 ID（role=tool 时关联对应 tool_call） |
| `toolCalls` | 助手发起的工具调用（role=assistant） |
| `events` | 过程展示事件（plan、thinking、toolResult 等） |
| `runStatus` | 运行状态（`COMPLETED` 或 `PAUSED_MAX_ITERATIONS`），仅助手消息 |
| `checkpointJson` | 最大迭代暂停时的协议级检查点 JSON，仅 `PAUSED_MAX_ITERATIONS` 时非空 |

`ChatMessageEntity` 表名 `chat_message`，索引 `idx_session_id` 和 `idx_timestamp`。`events` 字段保存前端历史展示需要的快照，不参与后续 LLM 上下文组装。`content` 和 `reasoning` 用 `TEXT` 类型。`run_status` 用 `VARCHAR(32)`，`checkpoint_json` 用 `LONGTEXT`，仅达到最大迭代上限时写入。下一轮 `prepare` 优先从检查点恢复协议级历史。

会话属于用户，但 Sandbox 资源按用户复用，不随会话删除销毁。

## 3. Agent 应用

### AgentAppEntity

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | `Long` | 主键，自增 |
| `userId` | `Long` | 应用所属用户 ID |
| `name` | `String(100)` | 应用名称 |
| `description` | `String(500)` | 应用描述 |
| `knowledgeBaseIds` | `Set<Long>` | 关联的知识库 ID 集合 |
| `enabledSkillIds` | `Set<String>` | 关联的 Skill ID 集合 |
| `createdAt` | `LocalDateTime` | 创建时间 |
| `updatedAt` | `LocalDateTime` | 更新时间 |

表名 `agent_app`，索引 `idx_app_user`。应用可关联多个知识库和多个 Skill。会话选择应用后，应用配置影响 `AgentTurnContext`：知识库描述注入工具、知识库增强上下文、Skill 过滤。

## 4. Sandbox

### UserSandboxEntity

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `userId` | `Long` | 用户 ID，同时作为主键 |
| `sandboxId` | `String(128)` | 用户当前绑定的沙箱 ID |
| `aioEndpoint` | `String(64)` | AIO 服务访问地址 |
| `deleted` | `Boolean` | 软删除标记，默认 false |
| `createdAt` | `LocalDateTime` | 绑定记录创建时间 |
| `updatedAt` | `LocalDateTime` | 绑定记录最后更新时间 |

表名 `user_sandbox`。一个用户对应一个沙箱，以 `userId` 为键。不健康记录会被软删除（`deleted=true`），启动恢复只恢复 `deleted=false` 且健康的记录。

`AioSandboxStore` 是内存 endpoint 缓存（`ConcurrentHashMap`），不是数据库实体。重启即失效，由 `UserSandboxEntity` 恢复。

### 5.1 KnowledgeBaseEntity

| 字段 | 类型 | 说明 |
| --- | --- | --- |

### 5.2 KnowledgeDocumentEntity

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | `Long` | 主键，自增 |
| `userId` | `Long` | 所属用户 ID |
| `kbId` | `Long` | 所属知识库 ID |
| `fileName` | `String` | 原始文件名 |

### 5.3 KnowledgeChunkEntity

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | `Long` | 主键，自增 |
| `documentId` | `Long` | 所属文档 ID |
| `userId` | `Long` | 所属用户 ID |
| `kbId` | `Long` | 所属知识库 ID |
| `chunkIndex` | `int` | 切片序号，从 0 开始 |
| `content` | `TEXT` | 切片正文 |
| `tokenCount` | `Integer` | 平均 token 数 |
| `startOffset` | `Integer` | 原文开始偏移 |
| `endOffset` | `Integer` | 原文结束偏移 |

表名 `knowledge_chunk`，索引 `idx_kc_document` 和 `idx_kc_user`。

### 5.4 向量

向量存储在 Milvus 集合 `knowledge_chunks_v3`，不在 MySQL。关闭 Milvus 时由 `NoopVectorStoreServiceImpl` 跳过写入和检索。

| --- | --- | --- |

## 8. 运行时模型（不持久化）

### 8.1 TodoState

运行时 Agent TodoState 是会话内内存状态，不持久化：

| 模型 | 说明 |
| --- | --- |
| `AgentTodoState` | 一次会话当前轮 todo 快照 |
| `AgentTodoItem` | id、title、status、successSignals、evidence、blocker、reason、batchIds |
| `AgentTodoStatus` | pending、in_progress、completed、blocked、cancelled |
| `AgentTodoUpdateResult` | 更新结果、警告和摘要 |

TodoState 主要服务最终回答门禁和多步任务闭环。

### 8.2 其他运行时模型

| 模型 | 说明 |
| --- | --- |
| `PlanResult` | 规划结果（plan 文本 + LlmUsage） |
| `AgentResponse` | 执行结果（finalAnswer、finalReasoning、steps、totalUsage） |
| `LlmUsage` | token 用量 record（prompt/completion/total/cacheHit） |
| `ToolDefinition` | 工具定义（name、description、parameters、sandboxType） |
| `ExecutionResult` | 执行结果（status、body、duration），status 为 SUCCESS/ERROR/TIMEOUT |

### 8.3 Skill

`Skill` 不是 JPA 实体，是技能元数据模型。来源分 local（本地仓库）和 sandbox（沙箱内扫描）两种。`localPath` 持有本地目录路径（用于 `FileSyncService.syncSkill` 推送），sandbox 来源的 `localPath` 为 null。

### 8.4 MCP 运行态

MCP 配置和运行态主要通过服务对象管理：

| 模型 | 说明 |
| --- | --- |
| `McpServerConfig` | server 配置 |
| `McpClientKey` | 系统级或用户级连接 key |
| `McpServerView` | 给前端和工具返回的 server 视图 |
| `RealMcpTool` | MCP 工具的 Tool 适配层 |

## 9. 实体关系总览

```text
UserEntity (1) ──< ConversationSessionEntity (N)
                      │
                      ├──< ChatMessageEntity (N)
                      ├── appId ──> AgentAppEntity
                      └── enabledSkillIds (Set<String>)

UserEntity (1) ──< UserSandboxEntity (1)     [userId 为主键]

UserEntity (1) ──< AgentAppEntity (N)
                      ├── knowledgeBaseIds (Set<Long>)
                      └── enabledSkillIds (Set<String>)

UserEntity (1) ──< KnowledgeBaseEntity (N)
                      └──< KnowledgeDocumentEntity (N)
                              ├──< KnowledgeChunkEntity (N)
                              └── vectors → Milvus (可选)

UserEntity (1) ──< TokenUsageEntity (N)
UserEntity (1) ──< WorkspaceDirectoryMemoryEntity (N)
```
