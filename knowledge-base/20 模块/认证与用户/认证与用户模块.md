# 认证与用户模块

## 1. 模块概览

本文说明注册、登录、当前用户识别和请求级用户上下文的实现。

当前实现里需要特别注意四点：

- 系统大多数业务数据按 userId 隔离：会话、应用、知识库、上传目录、用户 Sandbox、用户动态 MCP 配置等。
- `UserContext` 保存当前请求用户 ID，并带有 WebSearch、Planning 等请求级开关。
- 沙箱资源按用户持有，不按会话。
- 多租户边界是面试和代码审查重点。

## 2. 接口

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| POST | `/api/auth/register` | 注册 |
| POST | `/api/auth/login` | 登录 |
| GET | `/api/auth/me` | 获取当前用户信息 |

前端把 token、用户名和当前会话 ID 存在 `localStorage`。路由守卫通过 `/api/auth/me` 校验登录状态。

## 3. 用户实体

`UserEntity` 表名 `users`：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | `Long` | 主键，自增 |
| `username` | `String(64)` | 用户名，唯一索引 |
| `passwordHash` | `String(128)` | BCrypt 密码哈希 |
| `token` | `String(64)` | 访问令牌，唯一索引 |
| `createdAt` | `LocalDateTime` | 创建时间 |

`token` 是用户访问令牌，用于 API 鉴权。`regenerateToken` 可重新生成。

## 4. 用户上下文

`UserContext` 保存当前请求的用户 ID 和请求级开关：

| 字段/方法 | 说明 |
| --- | --- |
| `getCurrentUserId()` | 获取当前请求用户 ID |
| `isPlanningEnabled()` | 请求级规划开关 |
| `isWebSearchEnabled()` | 请求级网络搜索开关 |

Controller 和 Service 在处理会话、知识库、文件、Sandbox 时都应先确认数据属于当前用户。

## 5. 与 Sandbox 的关系

当前 Sandbox 资源按用户而不是会话持有。`SandboxServiceImpl` 通过会话解析 userId，再查找或创建用户级 Sandbox。沙箱视图 token 也只绑定 userId，并在代理请求时实时查当前 endpoint。

## 6. 数据隔离

| 数据 | 隔离 key | 校验方式 |
| --- | --- | --- |
| 会话 | userId | `validateSessionOwnership` |
| 知识库 | userId | `requireOwnedKnowledgeBase` |
| 文档 | userId | `requireOwnedDocument` |
| 文件 | userId | 用户级目录路径 |
| 沙箱 | userId | `UserSandboxEntity` |
| Agent 应用 | userId | `agentAppService` 查询 |
| Token 用量 | userId + sessionId | 记录时写入 |

## 7. 风险提醒

多租户边界是面试和代码审查重点。新增接口时要明确：

- 是否从当前用户上下文取 userId（不信任前端传入的 userId）。
- 是否校验目标资源属于该用户（通过 docId、kbId、sessionId、appId 越权是常见漏洞）。
- 是否可能通过文件路径穿越访问其他用户文件（`sanitizeFileName` 处理）。
- 用户动态 MCP 默认禁止 HTTP、私网和链路本地地址，降低 SSRF 风险。
