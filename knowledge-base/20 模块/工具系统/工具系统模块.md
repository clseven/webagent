# 工具系统模块

## 1. 模块概览

本文说明工具系统的接口设计、来源分类、上下文构建、并发调度和 Hook 边界。

当前实现里需要特别注意七点：

- 每个工具实现 `Tool` 接口，提供 OpenAI function calling 格式的 JSON Schema 定义，执行时返回字符串 observation。
- 工具异常时返回错误消息而非抛异常，确保 Agent 流程不中断。
- [[#ToolSideEffect]] 分 READ/WRITE/EXCLUSIVE 三类，并发调度器据此决定能否并行执行。
- 工具通过 `sandboxType` 声明支持的沙箱类型（ALL/AIO/COMMON），构建时按当前沙箱过滤。
- MCP 工具由 `McpClientToolProvider` 动态提供，与内置工具按原始名称去重。
- `web_search` 和 `run_subagent` 有独立开关，可按需过滤。
- `KnowledgeSearchTool` 的描述和默认知识库在构建时动态注入，对话结束后清理。

## 3. Tool 接口

### 3.1 设计原则

| 原则 | 说明 |
| --- | --- |
| 原子能力 | 每个工具实现一个具体能力（读文件、执行命令等） |
| OpenAI 兼容 | 工具定义遵循 OpenAI function calling 规范（JSON Schema） |
| 字符串返回 | 执行结果统一返回字符串，方便 LLM 理解 |
| 异常不抛 | 异常时返回错误消息而非抛异常，确保 Agent 流程不中断 |

### 3.2 接口方法

| 方法 | 说明 |
| --- | --- |
| `getDefinition()` | 返回 `ToolDefinition`（名称、描述、参数 schema），传给 LLM |
| `execute(sessionId, arguments)` | 执行工具，参数从 LLM 的 tool_calls 解析，返回结果字符串 |
| `getSideEffect()` | 声明副作用类型，默认 `EXCLUSIVE`（最保守） |

### 3.3 sandboxType

`ToolDefinition` 的 `sandboxType` 字段声明工具支持的沙箱：

| 值 | 含义 |
| --- | --- |
| `ALL` | 所有沙箱都可用 |
| `AIO` | 仅 AIO 沙箱可用 |
| `COMMON` | 仅普通沙箱可用 |

构建工具上下文时，`AgentToolContextService` 按当前沙箱类型过滤。当前项目统一使用 AIO 沙箱，`COMMON` 类工具会被过滤掉。

## 4. ToolSideEffect

[[#ToolSideEffect]] 是工具副作用类型枚举，供并发调度器决定能否并行执行。

| 类型 | 含义 | 调度规则 | 示例 |
| --- | --- | --- | --- |
| `READ` | 纯读，不改共享状态 | 可并发，与其他 READ 同时执行 | `web_search`、`read_file`、`grep` |
| `WRITE` | 改文件/沙箱状态 | 串行，且与 READ 互斥 | `write_file`、`file_replace` |
| `EXCLUSIVE` | 独占，改环境或有全局副作用 | 完全串行 | `execute_command`、`git` |

默认按最保守的 `EXCLUSIVE` 对待。只有确认无副作用的纯读类工具才显式覆盖为 `READ`。漏标不会导致数据竞争，只是失去并发收益；标错（把有副作用标成 READ）才危险。

并发执行受 `agent.hook.concurrent-tool-execution-enabled` 控制，出问题可置 false 退化为串行。

## 5. 工具上下文构建

入口方法：`AgentToolContextService.build`

### 5.1 构建流程

1. 判断当前会话沙箱类型（`isAioSandbox`），确定 `targetType`（AIO 或 COMMON）。
2. 按 `sandboxType` 过滤内置工具，只保留 `ALL` 或匹配 `targetType` 的。
3. 合并 MCP 动态工具（`mergeMcpTools`），按原始名称去重。
4. 过滤 `web_search` 工具（`filterWebSearchTool`），受 `UserContext.isWebSearchEnabled()` 控制。
5. 过滤 `run_subagent` 工具（`filterSubAgentTool`），受 `agent.sub-agent.enabled` 控制。
6. 配置 `KnowledgeSearchTool`（`configureKnowledgeSearchTool`），注入默认知识库 ID。
7. 构建工具定义列表（`buildToolDefinitions`），为 `KnowledgeSearchTool` 注入动态描述。
8. 返回 `AgentToolContext`。

### 5.2 MCP 工具合并

`mergeMcpTools` 的去重逻辑：

1. 收集内置工具的名称集合 `customNames`。
2. 遍历 MCP 工具，如果是 `RealMcpTool`，取 `getOriginalName()`。
3. 原始名称与内置工具冲突时，保留内置版本，跳过 MCP 版本。
4. 不冲突的 MCP 工具加入列表。

MCP 工具合并受 `agent.mcp.enabled` 控制，默认开启（`MCP_ENABLED=true`）。

### 5.3 KnowledgeSearchTool 动态配置

当应用关联知识库时：

1. `configureKnowledgeSearchTool` 找到 `KnowledgeSearchTool` 实例。
2. 取应用关联的第一个知识库 ID 作为 `defaultKbId`。
3. 调用 `setCurrentKbId(defaultKbId)` 设置默认检索范围。
4. `buildToolDefinitions` 为它生成带知识库描述的工具定义。

### 5.4 运行时清理

`clearRuntimeState` 在对话结束或异常时调用：

1. 检查 context 和 `knowledgeSearchTool` 是否为 null。
2. 调用 `clearCurrentKbId()` 清除默认知识库。
3. 调用 `clearDynamicDescription()` 清除动态描述。

清理避免会话级动态配置污染下一轮调用。同步路径在 `finally` 块清理，流式路径在 `doOnComplete` 和 `doOnError` 清理。

## 6. 工具来源

| 来源 | 说明 |
| --- | --- |
| 本地工具 | `src/main/java/com/example/sandbox/web/service/tool` 下的 Java 实现 |
| MCP 工具 | `McpClientToolProvider` 从已连接 MCP Server 读取工具定义，包装成 `RealMcpTool` |
| Skill 工具 | `skill_list`、`skill_activate`、`skill_reference` 负责渐进式披露 Skill 内容 |
| 运行时工具 | `todo_write`、`run_subagent` 影响 Agent 执行闭环，但仍遵守 Tool 接口 |

## 7. Hook 与 Tool 的边界

| 需求 | 用什么 |
| --- | --- |
| 拦截工具调用 | `PreToolUseHook`，返回非空字符串阻止执行 |
| 工具执行后注入消息 | `PostToolUseHook` |
| 最终回答前强制检查 | `StopHook` |

典型例子：`view_image` 工具只读取图片并放入 `ImageBuffer`；真正调用 Agnes 视觉模型的是 `ReactAgentHookService` 的 PostToolUseHook。工具本身只返回字符串，不直接追加 `ChatMessage`。

## 8. 风险与排查

| 现象 | 优先检查 | 说明 |
| --- | --- | --- |
| 工具没有出现 | `AgentToolContextService.build` 日志、sandboxType 过滤 | COMMON 类工具在 AIO 沙箱下会被过滤 |
| MCP 工具不出现 | `agent.mcp.enabled`、MCP Server 配置、`McpClientToolProvider` | 默认开启，检查 Server 是否连接 |
| web_search 不出现 | `UserContext.isWebSearchEnabled()` | 网络搜索开关关闭时移除 |
| run_subagent 不出现 | `agent.sub-agent.enabled` | 子代理开关关闭时移除 |
| knowledge_search 描述不对 | `configureKnowledgeSearchTool`、应用知识库关联 | 动态描述在构建时注入，检查 app.knowledgeBaseIds |
| 下一轮知识库检索范围错 | `clearRuntimeState` 是否调用 | 未清理会污染下一轮 |
| 并发工具执行异常 | `ToolSideEffect` 标注、`concurrent-tool-execution-enabled` | 检查是否有工具误标为 READ |
| 工具执行中断 Agent | 工具是否抛异常 | 工具应返回错误消息，不抛异常 |

## 9. 术语速查

本节用于 Obsidian 悬浮预览。阅读正文时，把鼠标停在带链接的术语上，可以快速看到这里的释义。

### Tool

`Tool` 是 Agent 可调用的能力单元接口。每个工具实现 `getDefinition`（返回 OpenAI 格式的 JSON Schema）、`execute`（执行并返回字符串结果）和 `getSideEffect`（声明副作用类型）。异常时返回错误消息而非抛异常，确保 Agent 流程不中断。

### ToolSideEffect

`ToolSideEffect` 是工具副作用类型枚举，分 READ（可并发）、WRITE（串行，与 READ 互斥）、EXCLUSIVE（完全串行）三种。并发调度器据此决定能否并行执行。默认 EXCLUSIVE 最保守，只有确认无副作用的纯读工具才标 READ。

### AgentToolContextService

`AgentToolContextService` 集中处理工具过滤、MCP 动态工具合并、知识库工具动态描述和工具状态清理。在每轮对话前构建工具上下文，对话结束后清理会话级运行状态。避免同步/流式对话入口重复理解工具细节。

### ToolDefinition

`ToolDefinition` 是工具定义，包含名称、描述、参数 schema 和 sandboxType。传给 LLM 让它知道有哪些工具可用以及如何调用。遵循 OpenAI function calling 规范。

### McpClientToolProvider

`McpClientToolProvider` 从已连接的 MCP Server 读取工具定义，包装成 `RealMcpTool`。MCP 工具与内置工具按原始名称去重，冲突时保留内置版本。受 `agent.mcp.enabled` 控制。

### RealMcpTool

`RealMcpTool` 是 MCP 工具的包装实现，`getOriginalName()` 返回 MCP Server 上的原始工具名，用于与内置工具去重。

### AgentToolContext

`AgentToolContext` 是单轮对话的工具上下文，包含过滤后的工具列表、工具定义列表、被配置的 `KnowledgeSearchTool` 和沙箱类型。由 `AgentToolContextService.build` 构建。
