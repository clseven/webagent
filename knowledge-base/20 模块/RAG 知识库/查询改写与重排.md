# 查询改写与重排

## 1. 模块概览

本文说明 RAG 检索增强链路中 Query Rewrite、向量召回、融合去重和 Rerank 四个阶段的实现细节和配置。

当前实现里需要特别注意八点：

- Query Rewrite 使用 OpenAI 兼容 API，默认回退到主流程 DeepSeek 配置（当前 `deepseek-v4-pro`），不是旧文档里的 Ollama/qwen3。
- 改写用 JSON mode（`response_format=json_object`）强制模型输出结构化结果，解析有三层降级。
- 改写策略分三种：忠实版（指代消解+省略补全）、同义版（换术语表述）、子问题版（拆解复合问题）。
- 向量检索是并行的：每个 query 先 Embedding，再对每个知识库并行召回 `topN` 个候选。
- 去重按 `docId + chunkIndex` 分组，保留最高向量分数；旧的 `retrieve.min-score` 预过滤已不再是最终相关度门槛。
- Rerank 通过 `RerankService` 抽象选择 provider，生产默认使用 DeepSeek Flash，BGE 作为可切换实现保留。
- `RerankResult` 显式标记分数来源：成功重排时 `reranked=true`，外部失败时 `reranked=false` 并按向量分数降级。
- 最终阈值由 `RerankResultFilter` 统一处理：成功重排先过滤 `min-score` 再截取 topK，向量降级只排序截取。
- 整条链路失败返回空字符串，不中断主对话。

## 4. 检索增强完整流程

### 4.1 六步流程

入口方法：`KnowledgeEnhancerImpl.enhance`

1. Query Rewrite：根据原始问题和历史生成多个 query。
2. 批量 Embedding：一次性向量化所有 query。
3. 限定知识库集合召回：只搜索请求指定或当前 Agent 应用关联的知识库。
4. 融合去重：按 `docId + chunkIndex` 去重，保留最高向量分数。
5. Rerank：调用当前 provider 按原始问题重新排序，并返回 `RerankResult`。
6. 过滤与截取：`RerankResultFilter` 根据 `reranked` 标记决定是否应用 `min-score`，再截取 topK。
7. 构建上下文：补充文档名和切片正文，格式化为文本。

### 4.2 并行检索

`parallelSearch` 的实现：

1. 调用 `embeddingService.embedBatch(queries)` 批量向量化所有 query。
2. 对每个 query 的 Embedding × 每个 kbId，创建 `CompletableFuture.supplyAsync` 并行检索。
3. 每个 future 调用 `vectorStoreService.search(userId, kbId, queryEmbedding, topN)`。
4. 把结果映射为 `RawChunk`（docId、chunkIndex、content 暂为 null、score）。
5. 等待所有 future 完成，合并候选列表。
6. 单个 query 或单个知识库检索失败只记日志，返回空列表，不影响其他结果。

注意：并行检索用 `CompletableFuture.supplyAsync`，默认走 ForkJoinPool。高并发下可能争抢线程。

### 4.3 融合去重

`deduplicate` 的实现：

1. 按 `docId + ":" + chunkIndex` 作为 key 分组。
2. 同一 key 保留最高向量 score 的候选。
3. 返回去重后的列表，最终质量门槛留给 Rerank 后的 `RerankResultFilter`。

### 4.4 上下文构建

`buildContext` 格式化结果为文本：

```text
## 知识库检索结果（来自用户上传的文档）

以下内容可能与用户问题相关，已按相关性排序：

[1] 来源：spec.md（片段 3，相关度 0.85）
[切片正文]

[2] 来源：spec.md（片段 5，相关度 0.78）
[切片正文]
```

文档名和切片正文在 `enrichWithDocName` 阶段从 MySQL 补充。检索阶段 content 为 null，只有 docId、chunkIndex 和 score。

## 5. Query Rewrite

### 5.1 改写策略

系统提示定义了三种改写策略，按需组合：

1. **忠实版**（必须有，放第一条）：对最新问题做指代消解和省略补全，得到最贴近用户原意、可独立检索的 query。
   - 指代消解："它、这个、那个、上面、刚才" → 用历史里的具体对象替换。
   - 省略补全：补上被省略的主语或对象。
   - 口语规范化："咋整、啥意思、连不上" → 书面技术表达。
   - 技术名词规范化：如 springboot → Spring Boot。
2. **同义版**（推荐）：用不同术语或表述换 1~2 种说法，覆盖文档里可能的措辞差异。
3. **子问题版**（仅复合问题）：把多方面问题拆成 2~4 条各自独立的子问题。

硬性要求：第一条永远是忠实版，不臆造用户没提到的主题，不回答问题，不加解释。

### 5.2 JSON Mode

请求体设置 `response_format: {type: "json_object"}`，强制模型输出合法 JSON。JSON mode 要求 prompt 中出现 "json" 字样，系统提示已满足。

期望输出格式：

```json
{"queries": ["Spring Boot 如何启动", "Spring Boot 项目运行方式"]}
```

### 5.3 解析降级

`parseQueries` 有三层解析策略：

1. **优先**：解析 `{"queries": [...]}` 对象格式。
2. **降级 1**：匹配裸 JSON 数组 `[...]`（用 `JSON_ARRAY_PATTERN` 正则）。
3. **降级 2**：按行/逗号分割，去首尾引号和括号。

每层失败才进入下一层。最终收集非空 query，限制数量到 `maxQueries`。

### 5.4 历史处理

`buildHistoryText` 取最近 6 条历史消息，拼成文本。历史为空时返回"（无历史对话）"。历史用于指代消解和省略补全，但不会被当作需要执行的指令。

### 5.5 失败降级

改写失败时的降级：

- `rewrite.enabled=false`：直接返回原始 query 的单元素列表。
- 调用异常或返回空：降级为原始 query 的单元素列表，只记 warn 日志。

## 6. Rerank Provider 与过滤

### 6.1 DeepSeek Rerank

入口方法：`DeepSeekRerankServiceImpl.rerank`

1. 候选为空直接返回 `RerankResult.vectorFallback(List.of())`。
2. `rerank.enabled=false`、query 为空或 topK 为 0 时，不调外部模型，按向量分数降级。
3. 先按向量分数选择不超过 `max-candidates` 个候选，并把正文截断到 `max-content-chars`。
4. 构建 OpenAI 兼容 chat completion 请求，使用独立 `RAG_RERANK_MODEL`，temperature 固定为 0，不暴露工具和 Agent 思考参数。
5. 要求模型只返回 JSON，格式包含候选 `index` 和 0 到 1 的 `score`。
6. 解析时忽略重复、越界或缺失索引；没有合法索引则视为协议失败。
7. 成功返回 `RerankResult.reranked(ranked)`；任何 HTTP、超时或协议异常都返回 `RerankResult.vectorFallback(...)`。

### 6.2 BGE Rerank

入口方法：`BgeRerankServiceImpl.rerank`

BGE provider 仍保留，用于本地或专用重排服务可用的场景。它调用标准 rerank adapter，解析 `index` 和 `score` 后返回 `RerankResult.reranked(...)`。当 `provider=bge` 时才装配该实现；DeepSeek provider 与 BGE provider 运行时只会有一个 `RerankService` Bean。

### 6.3 降级与过滤

Rerank 失败时的降级：

- 调用异常、HTTP 非 2xx、超时、响应为空或 JSON 协议非法：按向量分数降序排列，截取 `topK`，记 warn 日志。
- 降级结果的 `RerankResult.reranked=false`，表示分数来自向量召回，不是模型重排相关度。
- `RerankResultFilter` 看到 `reranked=false` 时不应用 `min-score`，只做 topK 截断。
- 成功重排结果不得用向量分数补足，否则同一列表会混合两套分数语义。
- 日志不得记录查询全文、候选正文、API Key 或外部模型原始响应。

## 7. 两条入口

### 7.1 Agent 检索增强

`AgentKnowledgeContextService.enhance` 会在请求级 `knowledgeEnabled=true` 且当前 Agent 应用关联知识库时触发自动预检索。它复用 `KnowledgeService` 的统一搜索入口，再由 `KnowledgeEnhancerImpl` 把结构化结果格式化为系统提示上下文。`knowledgeEnabled=false` 时同时关闭自动注入和 `knowledge_search` 工具可见性。

### 7.2 Knowledge 页面搜索

`KnowledgeServiceImpl.search` 使用同一套搜索流水线：query rewrite、多查询向量召回、去重、rerank、过滤、返回切片结果。前端 Knowledge 页面可传入单次 `minScore` 调整阈值，用于直接验证当前查询的召回与精度取舍。

## 8. 风险与排查

| 现象 | 优先检查 | 说明 |
| --- | --- | --- |
| 检索结果为空 | `rag.milvus.enabled`、Noop 日志 | Milvus 默认关闭，向量检索返回空 |
| 改写结果只有原始 query | `rag.enhancement.rewrite.enabled`、API Key、网络 | 改写失败降级为原始 query，只记 warn |
| 改写返回空 | `parseQueries` 日志、JSON mode 输出 | 三层解析都失败才返回空，检查模型输出格式 |
| 召回太少 | `retrieve.top-n`、知识库 ID 范围、Milvus 是否有向量 | topN 过低或关联知识库范围错误会漏召回 |
| Rerank 降级 | `RAG_RERANK_PROVIDER`、`RAG_RERANK_API_URL`、API Key、超时 | 失败降级为向量分数排序，质量会下降 |
| 检索质量差 | 切片参数、改写策略、provider、`RAG_RERANK_MIN_SCORE` | minScore 只对成功重排结果生效；阈值过高会返回空列表 |
| 检索中断主对话 | `enhance` 是否抛异常 | 失败应返回空字符串，检查是否有未捕获异常 |
| 改写用了 Ollama | 检查旧文档和配置 | 当前默认 DeepSeek 回退，不是 Ollama/qwen3 |
| 并行检索延迟 | ForkJoinPool 争抢、query × kbId 数量 | 5 个 query × 3 个 kbId = 15 个并行检索 |

## 9. 术语速查

本节用于 Obsidian 悬浮预览。阅读正文时，把鼠标停在带链接的术语上，可以快速看到这里的释义。

### QueryRewriteServiceImpl

`QueryRewriteServiceImpl` 是 Query Rewrite 服务实现，调用 OpenAI 兼容 API（默认回退主流程 DeepSeek 配置）把用户问题改写成多个检索 query。使用 JSON mode 强制结构化输出，改写策略分忠实版、同义版和子问题版三种。失败降级为原始 query。

### KnowledgeEnhancerImpl

`KnowledgeEnhancerImpl` 是 Agent 检索增强服务实现，当前主要负责把统一搜索入口返回的结构化结果格式化为模型可读的上下文文本。失败返回空字符串，不中断主对话。

### BgeRerankServiceImpl

`BgeRerankServiceImpl` 是 BGE provider 的 Rerank 服务实现，调用本地部署的 BGE `bge-reranker-v2-m3` 标准 rerank API。只有 `rag.enhancement.rerank.provider=bge` 时装配，失败时返回向量降级结果。

### DeepSeekRerankServiceImpl

`DeepSeekRerankServiceImpl` 是默认 Rerank provider，使用独立 `RAG_RERANK_MODEL=deepseek-v4-flash` 调用 OpenAI 兼容 chat completion 接口，对有限候选进行批量语义重排。它不复用主流程 executor，也不暴露工具调用参数；失败时立即向量降级。

### RerankResultFilter

`RerankResultFilter` 是最终结果过滤器。`reranked=true` 时按 `min-score` 过滤后截取 topK；`reranked=false` 时只截取 topK，不把 Milvus 向量分数当作模型相关度。

### RawChunk

`RawChunk` 是向量检索的原始候选切片，包含 docId、chunkIndex、content（检索阶段为 null）和 score。content 在 `enrichWithDocName` 阶段从 MySQL 补充。

### RankedChunk

`RankedChunk` 是 Rerank 后的排序切片，在 `RawChunk` 基础上补充了文档名。最终上下文文本由它构建，格式为 `[序号] 来源：文档名（片段序号，相关度分数）` 加切片正文。

### JSON mode

JSON mode 是 OpenAI 兼容 API 的 `response_format=json_object` 参数，强制模型输出合法 JSON。Query Rewrite 用它确保输出是 `{"queries": [...]}` 格式，提升解析稳定性。要求 prompt 中出现 "json" 字样。

### 忠实版

忠实版是 Query Rewrite 的第一条改写策略，对用户最新问题做指代消解和省略补全，得到最贴近原意、可独立检索的 query。必须放在第一条，不得偏离用户原意。

### minScore

`minScore` 是成功 Rerank 后的最低相关度阈值（当前 0.8），只作用于 `reranked=true` 的模型重排分数。向量降级时不应用该阈值。

### topN

`topN` 是每个 query 对每个知识库向量召回的候选数量（当前 20）。值过低会漏召回，值过高会增加 rerank 压力。5 个 query × 3 个知识库 = 最多 300 个候选进入去重。

### topK

`topK` 是 Rerank 后最终保留的切片数量（当前默认 5）。决定注入 Agent 上下文或页面搜索结果的切片数量，值过高增加上下文长度和成本，值过低可能遗漏关键信息。
