# 查询改写与重排

## 1. 模块概览

本文说明 RAG 检索增强链路中 Query Rewrite、向量召回、融合去重和 Rerank 四个阶段的实现细节和配置。

当前实现里需要特别注意七点：

- Query Rewrite 使用 OpenAI 兼容 API，默认回退到 DeepSeek 的 `deepseek-v4-flash`，不是旧文档里的 Ollama/qwen3。
- 改写用 JSON mode（`response_format=json_object`）强制模型输出结构化结果，解析有三层降级。
- 改写策略分三种：忠实版（指代消解+省略补全）、同义版（换术语表述）、子问题版（拆解复合问题）。
- 向量检索是并行的：每个 query 先 Embedding，再对每个知识库并行召回 `topN` 个候选。
- 去重按 `docId + chunkIndex` 分组，保留最高分，再过滤低于 `minScore` 的候选。
- Rerank 调用 BGE `bge-reranker-v2-m3`，失败时降级为向量分数排序。
- 整条链路失败返回空字符串，不中断主对话。

## 4. 检索增强完整流程

### 4.1 六步流程

入口方法：`KnowledgeEnhancerImpl.enhance`

1. Query Rewrite：根据原始问题和历史生成多个 query。
2. 并行向量检索：每个 query 先 Embedding，再对每个知识库并行检索 `topN` 个候选。
3. 融合去重：按 `docId + chunkIndex` 去重，保留最高分；过滤低于 `minScore` 的候选。
4. Rerank：调用 BGE Rerank 按原始问题重新排序。
5. 截取 topK：保留 `rerank.top-k` 个切片。
6. 构建上下文：补充文档名和切片正文，格式化为文本。

### 4.2 并行检索

`parallelSearch` 的实现：

1. 调用 `embeddingService.embedBatch(queries)` 批量向量化所有 query。
2. 对每个 query 的 Embedding × 每个 kbId，创建 `CompletableFuture.supplyAsync` 并行检索。
3. 每个 future 调用 `vectorStoreService.search(userId, kbId, queryEmbedding, topN)`。
4. 把结果映射为 `RawChunk`（docId、chunkIndex、content 暂为 null、score）。
5. 等待所有 future 完成，合并候选列表。
6. 单个 query 或单个知识库检索失败只记日志，返回空列表，不影响其他结果。

注意：并行检索用 `CompletableFuture.supplyAsync`，默认走 ForkJoinPool。高并发下可能争抢线程。

### 4.3 融合去重

`deduplicate` 的实现：

1. 按 `docId + ":" + chunkIndex` 作为 key 分组。
2. 同一 key 保留最高 score 的候选。
3. 过滤 score < `minScore` 的候选。
4. 返回去重后的列表。

### 4.4 上下文构建

`buildContext` 格式化结果为文本：

```text
## 知识库检索结果（来自用户上传的文档）

以下内容可能与用户问题相关，已按相关性排序：

[1] 来源：spec.md（片段 3，相关度 0.85）
[切片正文]

[2] 来源：spec.md（片段 5，相关度 0.78）
[切片正文]
```

文档名和切片正文在 `enrichWithDocName` 阶段从 MySQL 补充。检索阶段 content 为 null，只有 docId、chunkIndex 和 score。

## 5. Query Rewrite

### 5.1 改写策略

系统提示定义了三种改写策略，按需组合：

1. **忠实版**（必须有，放第一条）：对最新问题做指代消解和省略补全，得到最贴近用户原意、可独立检索的 query。
   - 指代消解："它、这个、那个、上面、刚才" → 用历史里的具体对象替换。
   - 省略补全：补上被省略的主语或对象。
   - 口语规范化："咋整、啥意思、连不上" → 书面技术表达。
   - 技术名词规范化：如 springboot → Spring Boot。
2. **同义版**（推荐）：用不同术语或表述换 1~2 种说法，覆盖文档里可能的措辞差异。
3. **子问题版**（仅复合问题）：把多方面问题拆成 2~4 条各自独立的子问题。

硬性要求：第一条永远是忠实版，不臆造用户没提到的主题，不回答问题，不加解释。

### 5.2 JSON Mode

请求体设置 `response_format: {type: "json_object"}`，强制模型输出合法 JSON。JSON mode 要求 prompt 中出现 "json" 字样，系统提示已满足。

期望输出格式：

```json
{"queries": ["Spring Boot 如何启动", "Spring Boot 项目运行方式"]}
```

### 5.3 解析降级

`parseQueries` 有三层解析策略：

1. **优先**：解析 `{"queries": [...]}` 对象格式。
2. **降级 1**：匹配裸 JSON 数组 `[...]`（用 `JSON_ARRAY_PATTERN` 正则）。
3. **降级 2**：按行/逗号分割，去首尾引号和括号。

每层失败才进入下一层。最终收集非空 query，限制数量到 `maxQueries`。

### 5.4 历史处理

`buildHistoryText` 取最近 6 条历史消息，拼成文本。历史为空时返回"（无历史对话）"。历史用于指代消解和省略补全，但不会被当作需要执行的指令。

### 5.5 失败降级

改写失败时的降级：

- `rewrite.enabled=false`：直接返回原始 query 的单元素列表。
- 调用异常或返回空：降级为原始 query 的单元素列表，只记 warn 日志。

## 6. BGE Rerank

### 6.1 调用

入口方法：`BgeRerankServiceImpl.rerank`

1. 候选为空直接返回空列表。
2. `rerank.enabled=false` 时，按向量分数降序排列，截取 `topK`，不调外部服务。
3. 提取候选的 content 作为 passages。
4. 构建请求：`{query, passages, topK, normalize=true}`。
5. POST 到 `api-url`（默认 `http://127.0.0.1:8002/rerank`）。
6. 解析响应的 `results`，每个 item 有 `index`（原候选位置）和 `score`（rerank 分数）。
7. 按 index 映射回原候选，构造 `RankedChunk`。

### 6.2 降级

Rerank 失败时的降级：

- 调用异常：按向量分数降序排列，截取 `topK`，记 warn 日志。
- 响应为空：同上降级。
- 降级时 `RankedChunk` 的 docName 为 null，后续由 `enrichWithDocName` 补充。

## 7. 两条入口

### 7.1 Agent 检索增强

`KnowledgeEnhancerImpl.enhance` 被 `AgentKnowledgeContextService.enhance` 调用，在 `AgentTurnContextService.prepare` 里注入系统提示。SOCIAL 轮次跳过。

### 7.2 Knowledge 页面搜索

`KnowledgeServiceImpl.search` 也使用类似流程：query rewrite、多查询向量召回、去重、rerank、返回切片结果。前端 Knowledge 页面可用于直接验证某个知识库的检索效果。

## 8. 风险与排查

| 现象 | 优先检查 | 说明 |
| --- | --- | --- |
| 检索结果为空 | `rag.milvus.enabled`、Noop 日志 | Milvus 默认关闭，向量检索返回空 |
| 改写结果只有原始 query | `rag.enhancement.rewrite.enabled`、API Key、网络 | 改写失败降级为原始 query，只记 warn |
| 改写返回空 | `parseQueries` 日志、JSON mode 输出 | 三层解析都失败才返回空，检查模型输出格式 |
| 召回太少 | `retrieve.top-n`、`min-score` | topN 过低或 minScore 过高会过滤掉有效候选 |
| Rerank 降级 | rerank 服务是否启动、`RAG_RERANK_URL` | 失败降级为向量分数排序，质量会下降 |
| 检索质量差 | 切片参数、改写策略、minScore | 切片太长太短、改写偏离原意、minScore 过高 |
| 检索中断主对话 | `enhance` 是否抛异常 | 失败应返回空字符串，检查是否有未捕获异常 |
| 改写用了 Ollama | 检查旧文档和配置 | 当前默认 DeepSeek 回退，不是 Ollama/qwen3 |
| 并行检索延迟 | ForkJoinPool 争抢、query × kbId 数量 | 5 个 query × 3 个 kbId = 15 个并行检索 |

## 9. 术语速查

本节用于 Obsidian 悬浮预览。阅读正文时，把鼠标停在带链接的术语上，可以快速看到这里的释义。

### QueryRewriteServiceImpl

`QueryRewriteServiceImpl` 是 Query Rewrite 服务实现，调用 OpenAI 兼容 API（默认 DeepSeek）把用户问题改写成多个检索 query。使用 JSON mode 强制结构化输出，改写策略分忠实版、同义版和子问题版三种。失败降级为原始 query。

### KnowledgeEnhancerImpl

`KnowledgeEnhancerImpl` 是检索增强服务实现，执行 Query Rewrite → 并行向量召回 → 融合去重 → Rerank → 截取 topK → 构建上下文的六步流程。失败返回空字符串，不中断主对话。

### BgeRerankServiceImpl

`BgeRerankServiceImpl` 是 Rerank 服务实现，调用本地部署的 BGE `bge-reranker-v2-m3` 标准 rerank API。失败时降级为向量分数排序，不中断检索链路。

### RawChunk

`RawChunk` 是向量检索的原始候选切片，包含 docId、chunkIndex、content（检索阶段为 null）和 score。content 在 `enrichWithDocName` 阶段从 MySQL 补充。

### RankedChunk

`RankedChunk` 是 Rerank 后的排序切片，在 `RawChunk` 基础上补充了文档名。最终上下文文本由它构建，格式为 `[序号] 来源：文档名（片段序号，相关度分数）` 加切片正文。

### JSON mode

JSON mode 是 OpenAI 兼容 API 的 `response_format=json_object` 参数，强制模型输出合法 JSON。Query Rewrite 用它确保输出是 `{"queries": [...]}` 格式，提升解析稳定性。要求 prompt 中出现 "json" 字样。

### 忠实版

忠实版是 Query Rewrite 的第一条改写策略，对用户最新问题做指代消解和省略补全，得到最贴近原意、可独立检索的 query。必须放在第一条，不得偏离用户原意。

### minScore

`minScore` 是向量召回的最低分数阈值（当前 0.5），低于此值的候选在去重阶段被过滤。值过高会过滤掉有效候选，值过低会引入噪声。

### topN

`topN` 是每个 query 对每个知识库向量召回的候选数量（当前 20）。值过低会漏召回，值过高会增加 rerank 压力。5 个 query × 3 个知识库 = 最多 300 个候选进入去重。

### topK

`topK` 是 Rerank 后最终保留的切片数量（当前 8）。决定注入 Agent 上下文的切片数量，值过高增加上下文长度和成本，值过低可能遗漏关键信息。
