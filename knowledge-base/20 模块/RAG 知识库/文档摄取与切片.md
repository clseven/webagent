# 文档摄取与切片

## 1. 文档概览

本文说明 RAG 知识库中文档上传后的处理链路：原始文件保存、文本解析、切片、[[#Embedding]]、向量写入、沙箱同步和预览转换。

当前实现里需要特别注意三点：

- 上传接口默认 [[#splitMode]]=`smart`，前端默认也选择智能切片。
- 全局配置里存在 `rag.chunk.size=500`、`rag.chunk.overlap=50`，但当前处理分支只有在 [[#splitMode]]=`custom` 且请求传入正数 [[#chunkSize]] 时才会走自定义切片。
- `smart` 模式使用 `TextSplitterServiceImpl` 内部硬编码阈值：段落长度超过 `1500` 字符后，按 `750/75` 二次切片。

## 2. 适用范围

### 2.1 本文覆盖

- Knowledge 页面上传文档到知识库。
- `RagController` 接收上传和替换请求。
- `KnowledgeServiceImpl` 保存文档、异步解析、切片和向量化。
- `KnowledgeDocumentProcessor` 异步处理链路与 `KnowledgeServiceImpl` 的删除、替换清理范围。
- 读取切片（`/document/{docId}/chunks`）和原始文件预览（`/document/{docId}/file`）接口，含沙箱文件按需恢复。
- `TextSplitterServiceImpl` 的 `smart` 与 `custom` 切片规则。
- 切片参数的配置来源、请求覆盖关系和落库字段。

### 2.2 本文不覆盖

- Query Rewrite、向量召回和 Rerank 的检索增强细节，见 [[查询改写与重排]]。
- Agent 如何调用 `knowledge_search`，见 [[Skill 与知识检索工具]]。
- Office 预览转换链路，见 [[Office 文件预览]]。

## 4. 上传入口与请求参数

### 4.1 上传文档

入口方法：`RagController.upload`

HTTP 接口：

```text
POST /api/rag/bases/{kbId}/documents/upload
```

请求参数：

| 参数 | 类型 | 必填 | 默认值 | 说明 |
| --- | --- | --- | --- | --- |
| `kbId` | `Long` | 是 | 无 | 目标知识库 ID，来自路径 |
| `file` | `MultipartFile` | 是 | 无 | 上传的原始文档 |
| [[#splitMode]] | `String` | 否 | `smart` | 支持 `smart` 和 `custom` |
| [[#chunkSize]] | `Integer` | 否 | 无 | 仅 `custom` 模式且大于 0 时生效 |
| [[#overlap]] | `Integer` | 否 | 无 | 仅 `custom` 模式时生效；未传或小于等于 0 时按 0 处理 |

### 4.2 替换文档

入口方法：`RagController.replaceDocument`

HTTP 接口：

```text
PUT /api/rag/document/{docId}/replace
```

替换文档复用同一套参数和异步处理流程。替换前的清理范围（`KnowledgeServiceImpl.replaceDocument`）：

1. 删除旧 Milvus 向量（`vectorStoreService.deleteByDocId`，失败仅告警不阻断）。
2. 删除旧 MySQL 切片（`chunkRepository.deleteByDocumentId`）。
3. 删除沙箱内旧文件——**使用旧文件名**，因为 `syncToSandbox` 写入时用的就是原始文件名；若新旧文件名不同，删的是旧名那份，新名文件由异步流程重新同步。
4. 本地旧文件路径与新路径不同时，`Files.deleteIfExists` 清掉旧本地文件。
5. 重置文档元数据：`chunkCount=0`、[[#totalTokens]]=`null`、[[#sandboxSynced]]=`false`、`errorMsg=null`、`status=PENDING`，保留 `id` 和 `createdAt`。

清理完成后，按新文件重新写入规范路径，再走与上传一致的异步解析、切片、[[#Embedding]] 和向量写入流程。

注意：替换**不做同名重复检查**，新文件名若与同知识库下其他文档重名不会被拦截（与上传 4.1 的行为不对称），可能造成同库下出现两条同名文档记录。

### 4.3 支持的文件类型与上传校验

文档解析由 `DocumentParserServiceImpl` 基于 Apache Tika 实现，理论上支持 PDF/DOCX/PPTX/XLS/HTML/图片等 700+ 格式，解析失败会抛 `RuntimeException` 并在异步阶段把文档置为 `FAILED`。

当前上传链路的校验现状：

- **没有文件大小上限**：`MultipartFile` 体积只受 Spring/网关层 `multipart.max-file-size` 限制，业务代码不拦截。
- **没有文件类型白名单**：任何扩展名都能上传，`getFileExtension` 在文件名无扩展名时默认记为 `txt`。
- **仅做文件名清洗**：`workspaceStorage.sanitizeFileName` 处理路径穿越和非法字符，不做类型判断。

风险与建议：

- 大文件会直接进入 Tika 解析，CPU 和内存占用仍可能较高；虽然异步处理已使用有界线程池，但大文件会长时间占用工作线程和队列容量，批量上传时仍可能拖慢后续文档处理。
- 图片、音视频等无法提取有效文本的格式，解析后正文可能为空，最终落在 `文档内容为空或无法切分` 的 `FAILED` 状态。
- 建议在上传入口加类型白名单和大小上限，提前拦截无意义上传，避免浪费解析和 [[#Embedding]] 成本。

### 4.4 读取切片与原文件

上传和异步处理完成后，前端通过两个读侧接口获取切片数据和原始文件预览。

#### 4.4.1 获取文档切片

入口方法：`RagController.listChunks`

```text
GET /api/rag/document/{docId}/chunks
```

返回该文档全部切片，字段为 `id`、`chunkIndex`、`content`、[[#tokenCount]]、[[#startOffset]]、[[#endOffset]]。这是 [[#5.3 位置标注的局限性]] 里前端做原文高亮定位的数据来源——位置错位的实际影响面就在这个接口的消费端。

#### 4.4.2 获取原始文件（预览）

入口方法：`RagController.getFile` → `KnowledgeServiceImpl.getPreviewContent`

```text
GET /api/rag/document/{docId}/file
```

从用户沙箱读取原始文件返回给前端预览。可转换的 Office 文件走 [[Office 文件预览]] 链路，其余按 `mediaType` 直接返回字节。

前置条件：`getPreviewContent` 开头会取 `sandboxClientFactory.getAioClientByUserId(userId)`，若用户**没有沙箱**（返回 `null`）直接抛 `用户沙箱不可用，请稍后重试`，既不走按需恢复、也不从本地读。因此下文「本地文件还在就能自愈」只在**用户有沙箱**时成立。

关键点：`getPreviewContent` 在读沙箱前会调 `ensureSandboxFile` 做**按需恢复**——若沙箱中目标路径文件不存在，会自动 `mkdir -p` 父目录并从本地规范路径重新 `writeBytes` 推一份到沙箱。因此即使 [[#sandboxSynced]]=`false`（异步同步阶段失败）或沙箱文件事后被清掉，只要本地文件还在，访问该接口仍能自愈补上沙箱文件并正常预览。排查「文件无法预览」时不能只看 `sandboxSynced`，要确认本地 `storagePath` 是否存在。

`getPreviewContent` 和 `getFileContent` 在读沙箱前都会先调 `migrationService.ensureCanonicalFile(document)`，把历史旧路径文档规范化到当前用户级存储路径（兼容早期 `rag.storage.path` 时代的落盘位置），是历史数据迁移的兜底入口。

两者关键差异：`getPreviewContent` 会调 `ensureSandboxFile` 按需补传沙箱文件（见 4.4.2）；`getFileContent` **不**调 `ensureSandboxFile`，沙箱文件缺失即抛 `沙箱文件不存在或为空`，不会从本地恢复。排查「文件无法读取」时先确认走的是哪个接口——前者能自愈，后者不能。

## 5. 切片模式

### 5.1 `smart` 智能切片

`smart` 是默认模式，入口是 `TextSplitterServiceImpl.splitSmartWithPosition`。

处理规则：

1. 按两个或更多换行符切成自然段落。
2. 段落为空时跳过。
3. 段落长度不超过 `SMART_MAX_LENGTH=1500` 时，整段保留为一个 chunk。
4. 段落长度超过 `1500` 时，调用 `splitCustom` 做二次切分。
5. 二次切分参数固定为 `SMART_CHUNK_SIZE=750`、`SMART_OVERLAP=75`。
6. 切分完成后通过 [[#attachPositions]] 回填原文中的 [[#startOffset]] 和 [[#endOffset]]。

适合场景：

- Markdown、TXT、解析后段落结构比较完整的文档。
- 普通项目文档、说明文、面试材料等以自然段落为主的内容。
- 希望尽量保留段落完整性，而不是强制按固定长度切开。

注意事项：

- `smart` 模式不使用请求里的 [[#chunkSize]] 和 [[#overlap]]。
- `smart` 模式不直接使用 `rag.chunk.size=500` 和 `rag.chunk.overlap=50`。
- 如果文档解析后几乎没有空行，很多内容可能会被当成超长段落，再按 `750/75` 二次切分。

### 5.2 `custom` 自定义切片

`custom` 模式入口是 `TextSplitterServiceImpl.splitCustomWithPosition`。

后端进入该模式需要同时满足：

```text
splitMode == "custom"
chunkSize != null
chunkSize > 0
```

处理规则：

1. 先按两个或更多换行符切成段落。
2. 尝试把段落拼接到当前 chunk。
3. 如果拼接后长度超过 [[#chunkSize]]，先保存当前 chunk。
4. 保存当前 chunk 后，取尾部 [[#overlap]] 字符作为下一片的开头上下文。
5. 如果单个段落本身超过 [[#chunkSize]]，调用 [[#splitLongText]] 按中英文句号、问号、感叹号进一步切分。
6. 切分完成后通过 [[#attachPositions]] 回填原文偏移。

适合场景：

- PDF 或 Office 解析后段落过长，需要主动控制 chunk 长度。
- 检索召回片段太长、噪声太多，需要缩短 chunk。
- 问答需要跨句上下文，希望通过 `overlap` 保留前后衔接。

注意事项：

- [[#overlap]] 未传、为 `null` 或小于等于 0 时，后端按 `0` 处理。
- [[#overlap]] 过大时会增加重复内容、[[#Embedding]] 成本和检索噪声。
- 当前后端没有显式限制 `overlap < chunkSize`，前端输入框用 `max=chunkSize` 做了界面限制。
- [[#splitLongText]] 的断句正则 `(?<=[。！？.!?])\s*` 会把英文句号也当作断点，`e.g.`、`Dr.`、`3.14`、`Section 1.2` 等英文缩写和版本号会被误切。
- 单个句子本身超过 [[#chunkSize]] 时，[[#splitLongText]] 会先保存已积累内容，再调用 `splitHard` 按 `chunkSize` 做字符级硬截断兜底（带 [[#overlap]] 滚动窗口），保证单片不超过 [[#chunkSize]]。
- 超长段落被二次切分后，会保留最后一片的尾部 [[#overlap]] 作为下一片开头，避免段落间上下文断裂。

### 5.3 位置标注的局限性

切片完成后统一由 [[#attachPositions]] 回填 [[#startOffset]] 和 [[#endOffset]]，策略是 `text.indexOf(chunk, searchStart)` 整体匹配，找不到时走「累积偏移」fallback。这带来一个已知局限：

- 只要切片带 overlap（`custom` 的 [[#overlap]]，或 `smart` 二次切片的 `SMART_OVERLAP`），chunk 文本会把「上一片尾部 + 下一片内容」拼在一起，这段拼接字符串在原文中**并不连续出现**，`indexOf` 必然失败，只能走 fallback。
- fallback 的 [[#startOffset]] 只是「上一片 end + chunk 长度」的兜底，**并非真实原文位置**，并且会随失败次数累积偏差，越往后偏移越大。
- 实际影响：纯自然段落切片（无 overlap、无二次切）位置准确；一旦出现 overlap 或超长段落二次切，前端基于偏移量的高亮定位会错位。

## 6. 文档处理流程

### 6.1 同步上传阶段

入口：`KnowledgeServiceImpl.upload`

1. 校验知识库属于当前用户。
2. 清洗原始文件名。
3. 用 step 2 清洗后的文件名，检查同一用户、同一知识库下是否已有同名文件（忽略大小写）。若已存在，**直接抛出 `DuplicateFileException` 阻断上传**：不会创建文档记录，也不会触发异步处理。异常里带 `existingDocId`，前端可据此引导用户走替换流程。
4. 创建 [[#KnowledgeDocumentEntity]]，初始状态来自实体默认值 `PENDING`。
5. 保存 `userId`、`kbId`、`fileName`、`fileType`、`fileSize`、[[#splitMode]]、[[#chunkSize]]、[[#overlap]]。
6. 将原始文件写入用户级知识库目录。
7. 更新 `storagePath`。
8. 调用 `KnowledgeDocumentProcessor.processDocumentAsync` 异步处理（跨 Bean 调用，`@Async` 通过 Spring 代理生效）。
9. 立即向前端返回文档元数据，此时状态为 `PENDING`，文档在后台异步处理。

### 6.2 异步处理阶段

入口：`KnowledgeDocumentProcessor.processDocumentAsync`（从 `KnowledgeServiceImpl` 拆出，避免同类自调用导致 `@Async` 失效）

```text
PENDING
  -> PROCESSING
    -> READY
    -> FAILED
```

处理步骤：

1. 根据 `docId` 查询文档记录。
2. 将状态更新为 `PROCESSING`。
3. 调用 `DocumentParserService` 解析原始文件正文。
4. 根据 [[#splitMode]]、[[#chunkSize]]、[[#overlap]] 选择切片策略。
5. 如果没有生成任何切片，状态改为 `FAILED`，错误信息为 `文档内容为空或无法切分`。
6. 调用 `EmbeddingService.embedBatch` 批量生成向量，并获取真实 [[#promptTokens]]。
7. 将平均 token 数写入每个 [[#KnowledgeChunkEntity]] 的 [[#tokenCount]]。
8. 保存每个切片的正文、序号、用户、知识库、文档 ID 和原文偏移。
9. 调用 `VectorStoreService.insertBatch` 写入向量库。
10. 调用 `syncToSandbox` 同步文件到用户 Sandbox，用于预览。
11. 如果沙箱同步成功，异步触发 Office 预转换。
12. 更新 `chunkCount`、[[#totalTokens]]、[[#sandboxSynced]] 和状态 `READY`。
13. 任意异常会将状态改为 `FAILED`，并把异常消息写入 `errorMsg`。

注意事项：

- `processDocumentAsync` 已拆到独立 Bean `KnowledgeDocumentProcessor`，通过 `@Async("knowledgeTaskExecutor")` 在 `AsyncConfig` 定义的有界线程池（core=2、max=8、queue=100、`CallerRunsPolicy`）上执行，真正异步、不阻塞上传请求线程；队列满时回退为调用线程执行，起到限流作用。历史问题：原本 `@Async` 写在 `KnowledgeServiceImpl` 内部、由同类 `upload`/`replaceDocument` 自调用，Spring 代理不生效，实际是同步执行；拆 Bean 后才真正异步。
- 上传/替换接口返回时文档仍是 `PENDING`/`PROCESSING`，前端 `Knowledge.js` 在上传完成后启动轮询（每 3s 调一次 `loadDocuments`），直到所有文档稳定为 `READY`/`FAILED` 才停止；组件卸载时清理定时器。
- 异步方法**没有事务包裹**，每步 `save` 都是独立事务。如果在「切片已写 MySQL、向量已写 Milvus」之后、状态置 `READY` 之前抛异常，catch 只会把状态改成 `FAILED`，**已写入的切片和向量不会被回滚或清理**，形成脏数据残留，直到下次 `replaceDocument` 或 `deleteDocument` 才会清掉。排查 `FAILED` 文档时需留意是否残留切片和向量。
- **`READY` 不等于沙箱已同步**：`syncToSandbox` 失败时只把 [[#sandboxSynced]] 置 `false`，**不会把状态改成 `FAILED`**，文档依然落到 `READY`。因此一个 `READY` 文档可能预览不可用——但只要本地 `storagePath` 还在，访问 `/document/{docId}/file` 时 `ensureSandboxFile` 会按需补传，详见 [[#4.4 读取切片与原文件]]。
- **`replaceDocument` 的 `@Async` 启动时序风险**：`upload` 没有 `@Transactional`，`save` 随即提交，异步线程能查到 `PENDING` 记录；但 `replaceDocument` 带 `@Transactional`，异步任务提交发生在事务方法内部。如果异步线程早于外层事务提交读取文档，理论上可能读到更新前的旧记录。当前代码未用 `TransactionSynchronizationManager` 注册 afterCommit 回调，遇到替换后状态或文件名异常时可优先排查这里。

### 6.3 删除文档与清理

入口：`KnowledgeServiceImpl.deleteDocument`（`DELETE /api/rag/document/{docId}`，带用户权限校验的版本会先 `requireOwnedDocument`）

清理范围，按顺序执行：

1. 删除 Milvus 向量（`vectorStoreService.deleteByDocId`，失败仅告警不阻断）。
2. 删除 MySQL 切片（`chunkRepository.deleteByDocumentId`）。
3. 删除本地原始文件（`Files.deleteIfExists(storagePath)`，失败仅告警）。
4. 删除沙箱内文件（`deleteFromSandbox`，用原始文件名拼路径，`rm -f` 执行，失败仅告警）。
5. 删除文档记录。

这是 6.2 提到的脏数据残留的主要清理途径。注意 Milvus、本地文件、沙箱文件任一删除失败都**只记日志、不回滚**，删除后可能残留少量孤儿数据，排查时需分别核对 Milvus 集合、本地目录和沙箱路径。`deleteKnowledgeBase` 会逐个调用本方法清掉知识库下所有文档，再 `deleteByKbId` 清理 Milvus。

### 6.4 用户级批量清理

入口：`KnowledgeServiceImpl.deleteAllByUser(userId)`（账号注销等场景，目前无 Controller 直接暴露，由服务层调用）

清理范围：

1. 删除该用户全部 Milvus 向量（`vectorStoreService.deleteByUserId`，失败仅告警）。
2. 删除该用户全部 MySQL 切片（`chunkRepository.deleteByUserId`）。
3. `Files.walk` 反向删除本地 `knowledgeRoot(userId)` 整个目录。
4. 删除该用户全部文档记录和知识库记录。

已知不一致：本方法**不调用 `deleteFromSandbox`**，也不逐个清理沙箱文件，因此用户级删除会在沙箱（`/home/gem/knowledge/...`）下残留孤儿文件——与 6.3 的 `deleteDocument`（逐个删沙箱）行为不对称。排查用户注销后的沙箱占用时需留意这点，必要时手动清理沙箱目录。

## 8. 配置与代码关系

### 8.1 配置绑定

`RagConfigProperties` 使用：

```java
@ConfigurationProperties(prefix = "rag")
```

绑定 `application.yml` 的 `rag.*` 配置。

当前 `Chunk` 内部默认值：

```java
private int size = 500;
private int overlap = 50;
```

当前 `application.yml` 覆盖值：

```yaml
rag:
  chunk:
    size: 500
    overlap: 50
```

### 8.2 当前实现限制

当前代码中没有发现 `config.getChunk().getSize()` 或 `config.getChunk().getOverlap()` 被上传处理链路直接调用。因此：

- `rag.chunk.size` 和 `rag.chunk.overlap` 是已定义配置项。
- 实际上传切片默认模式是 `smart`。
- `custom` 模式的参数主要来自前端或调用方提交的 [[#chunkSize]]、[[#overlap]]。
- 如果调用方传 [[#splitMode]]=`custom` 但不传 [[#chunkSize]]，后端会回退到 `smart`。

## 9. 调参建议

### 9.1 选择切片模式

| 场景 | 推荐模式 | 推荐原因 |
| --- | --- | --- |
| 文档有清晰段落和标题 | `smart` | 保留自然段落，减少语义被切断 |
| PDF 解析后大段文本没有换行 | `custom` | 可以主动控制 chunk 长度 |
| 检索片段经常包含太多无关内容 | `custom`，降低 [[#chunkSize]] | 缩短片段，减少噪声 |
| 回答经常缺少前后文 | `custom`，提高 [[#overlap]] | 保留相邻片段上下文 |
| [[#Embedding]] 成本过高 | 适当提高 [[#chunkSize]]，降低 [[#overlap]] | 减少切片数量和重复内容 |

### 9.2 参数经验值

| 参数组合 | 适合内容 | 风险 |
| --- | --- | --- |
| `chunkSize=500, overlap=50` | 短文档、FAQ、说明类文本 | 片段偏短，可能缺上下文 |
| `chunkSize=750, overlap=75` | 通用项目文档，接近当前前端默认值 | 平衡成本和上下文 |
| `chunkSize=1000, overlap=100` | 长文档、技术方案、接口说明 | 单片更长，召回噪声可能上升 |
| `chunkSize=1500, overlap=150` | 大段叙述性文档 | 切片少但单片成本更高 |

## 10. 排查 Checklist

| 现象 | 优先检查 | 说明 |
| --- | --- | --- |
| 上传后一直显示处理中 | `knowledge_document.status`、异步日志 | 前端每 3s 轮询刷新；若长期不变，说明后端卡在解析、[[#Embedding]]、向量写入或沙箱同步 |
| 状态为 `FAILED` | `knowledge_document.error_msg`、`knowledge_chunk`、Milvus | 异步异常会写入该字段；注意失败前可能已写入切片和向量，需确认是否残留脏数据，必要时走替换或删除清理 |
| 上传提示文件重复 | 同一知识库下同名文件 | 同步阶段抛 `DuplicateFileException`，不会进入异步处理，需改走替换接口 |
| 切片数量为 0 | 文档解析结果、`errorMsg` | 常见原因是文档为空、解析失败或无法切分 |
| 切片过多 | [[#splitMode]]、[[#chunkSize]]、[[#overlap]] | `chunkSize` 过小或 `overlap` 过大都会增加切片数量 |
| 切片太长 | 是否使用 `smart`，原文段落是否过长 | `smart` 会保留 1500 字以内的完整段落 |
| 检索不到内容 | `rag.milvus.enabled`、向量写入日志、知识库范围 | [[#Milvus]] 默认关闭时，向量检索会降级为空结果 |
| 文件无法预览 | 本地 `storagePath`、[[#sandboxSynced]]、Sandbox 文件路径 | 预览依赖沙箱文件；`/file` 接口的 `ensureSandboxFile` 会在沙箱缺失时从本地补传，所以优先确认本地 `storagePath` 是否存在，`sandboxSynced=false` 不代表一定无法预览 |
| 文档重复上传报错 | 同一知识库下同名文件 | `KnowledgeServiceImpl.upload` 会按文件名忽略大小写检查重复 |
| 用户注销后沙箱占用未释放 | 沙箱 `/home/gem/knowledge/...` 目录 | `deleteAllByUser` 不清理沙箱文件（见 6.4），会留孤儿文件，需手动清理 |

## 11. 扩展建议

如果后续要让 `rag.chunk.size` 和 `rag.chunk.overlap` 真正成为后端默认切片参数，建议调整 `KnowledgeServiceImpl.processDocumentAsync`：

1. 当 [[#splitMode]]=`custom` 且 [[#chunkSize]] 为空时，读取 `config.getChunk().getSize()`。
2. 当 [[#splitMode]]=`custom` 且 [[#overlap]] 为空时，读取 `config.getChunk().getOverlap()`。
3. 对 [[#chunkSize]] 和 [[#overlap]] 做边界校验，例如 `chunkSize > 0`、`0 <= overlap < chunkSize`。
4. 将最终生效参数写回 [[#KnowledgeDocumentEntity]]，避免落库字段是 `null` 但实际使用了默认值。
5. 如果希望 `smart` 也可配置，应把 `SMART_MAX_LENGTH`、`SMART_CHUNK_SIZE`、`SMART_OVERLAP` 从硬编码常量迁移到配置项。

## 12. 术语速查

本节用于 Obsidian 悬浮预览。阅读正文时，把鼠标停在带链接的术语上，可以快速看到这里的解释。

### attachPositions

`attachPositions` 是切片后的定位补偿步骤。

切片函数先得到一组 chunk 文本，再把每个 chunk 映射回解析后原文中的位置，补上 `startOffset` 和 `endOffset`。这样前端预览时可以知道某个切片对应原文的哪一段。

### startOffset

`startOffset` 是切片在解析后原文中的开始字符偏移。

例如 `startOffset=1200` 表示这个 chunk 大约从全文第 1200 个字符开始。它不是文件字节位置，而是解析后文本里的字符位置。

### endOffset

`endOffset` 是切片在解析后原文中的结束字符偏移。

它通常和 `startOffset` 一起使用，用于前端定位、切片预览和原文范围高亮。

### splitMode

`splitMode` 是上传文档时选择的切片模式。

当前主要有两种值：`smart` 表示智能切片，优先保留自然段落；`custom` 表示自定义切片，使用调用方传入的 `chunkSize` 和 `overlap`。

### chunkSize

`chunkSize` 是自定义切片模式下的目标切片长度。

当前实现按 Java 字符串长度处理，可以近似理解为字符数，不是严格 token 数。值越小，切片越碎；值越大，单片上下文越多，但检索噪声和 Embedding 成本也可能增加。

### overlap

`overlap` 是相邻切片之间保留的重复文本长度。

它的作用是减少语义被切断的问题。值过小可能缺上下文，值过大则会带来重复内容、更多向量和更高检索噪声。

### splitLongText

`splitLongText` 是自定义切片里的兜底长文本切分方法。

当单个段落超过 `chunkSize` 时，它会尝试按中英文句号、问号、感叹号等自然断句位置继续切开。

### Embedding

Embedding 是把文本切片转换成向量的过程。

系统会把每个 chunk 发送给 Embedding 服务，拿到向量后才能写入 Milvus 做相似度检索。

### Milvus

Milvus 是当前项目使用的向量数据库。

开启后，切片向量会写入集合 `knowledge_chunks_v3`；关闭时，系统仍然保存文档和切片正文，但 `VectorStoreService` 退化为 Noop——**写入侧 `insertBatch` 也静默不写**，检索返回空结果。排查「切片已落库但检索不到」时需先确认 `rag.milvus.enabled`。

### promptTokens

`promptTokens` 是 Embedding API 返回的输入 token 数。

当前异步处理会把整批切片的 `promptTokens` 作为文档 `totalTokens`，再用平均值估算每个 chunk 的 `tokenCount`。

### tokenCount

`tokenCount` 是单个切片记录里的 token 数字段。

当前不是每片单独精确计算，而是用文档总 token 数除以切片数量得到平均值。

### totalTokens

`totalTokens` 是文档级 token 总数。

它来自 Embedding 服务返回的 `promptTokens`，可用于估算向量化成本和判断文档规模。

### sandboxSynced

`sandboxSynced` 表示原始知识库文件是否已同步到用户 Sandbox。

它主要影响文件预览能力。为 `false` 时，文档可能已经完成解析和切片，但前端无法通过 Sandbox 路径打开原始文件预览。

### KnowledgeDocumentEntity

`KnowledgeDocumentEntity` 是知识库文档表实体。

它记录文档级元数据，例如文件名、存储路径、切片模式、切片数量、处理状态、失败原因和沙箱同步状态。

### KnowledgeChunkEntity

`KnowledgeChunkEntity` 是知识库切片表实体。

它记录每个 chunk 的正文、序号、所属文档、所属知识库、估算 token 数，以及在原文中的开始和结束偏移。
