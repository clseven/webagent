# 文件与工作区模块

## 1. 模块概览

本文说明用户级文件存储、沙箱文件同步、上传/下载/删除和知识库文件迁移的实现细节。

当前实现里需要特别注意六点：

- 文件存储是用户级的，不再按会话隔离。路径结构为 `uploads/users/{userId}/uploads/` 和 `uploads/users/{userId}/knowledge/{kbId}/`。
- `UserWorkspaceStorageService` 管理本地路径，`sanitizeFileName` 处理路径穿越和非法字符。
- 沙箱内路径是固定的：`/home/gem/uploads/` 和 `/home/gem/knowledge/{kbId}/`。
- `FileSyncService` 在沙箱创建时把用户文件和知识库文件同步到新沙箱。
- `KnowledgeFileMigrationService` 把历史旧路径文件迁移到当前用户级规范路径。
- Office 文件上传后触发异步预转换，便于后续预览。

## 4. 用户级目录结构

### 4.1 本地存储

```text
uploads/users/{userId}/
├─ uploads/{fileName}           # 用户上传的普通文件
└─ knowledge/{kbId}/{fileName}  # 知识库原始文件
```

`UserWorkspaceStorageService` 的路径构造方法：

| 方法 | 返回路径 | 用途 |
| --- | --- | --- |
| `userRoot(userId)` | `basePath/users/{userId}` | 用户根目录 |
| `uploadRoot(userId)` | `basePath/users/{userId}/uploads` | 上传文件根目录 |
| `knowledgeRoot(userId)` | `basePath/users/{userId}/knowledge` | 知识库文件根目录 |
| `uploadFile(userId, fileName)` | `basePath/users/{userId}/uploads/{safeName}` | 具体上传文件 |
| `knowledgeFile(userId, kbId, fileName)` | `basePath/users/{userId}/knowledge/{kbId}/{safeName}` | 具体知识库文件 |

### 4.2 沙箱存储

```text
/home/gem/
├─ uploads/{fileName}           # 用户上传文件
├─ workspace/                   # Agent 工作目录
├─ output/                      # 生成产物
├─ skills/                      # Skill 文件
├─ tools/                       # 工具脚本
├─ knowledge/{kbId}/{fileName}  # 知识库原始文件和预览文件
└─ .runtime/browser-agent/      # Browser Agent 运行时（隐藏）
```

沙箱路径映射方法：

| 方法 | 返回路径 |
| --- | --- |
| `uploadSandboxPath(fileName)` | `/home/gem/uploads/{safeName}` |
| `knowledgeSandboxPath(kbId, fileName)` | `/home/gem/knowledge/{kbId}/{safeName}` |

### 4.3 sanitizeFileName

`sanitizeFileName` 处理文件名安全：

1. null 检查，为空抛 `IllegalArgumentException`。
2. 反斜杠转正斜杠（`replace('\\', '/')`）。
3. 取最后一段（`lastIndexOf('/') + 1`），去掉路径前缀。
4. trim 后检查空、`.`、`..`，非法抛异常。

注意：`sanitizeFileName` 只做文件名清理，不做类型判断。任何扩展名都能通过。

## 5. 文件操作

### 5.1 上传

入口方法：`FileUploadController`

1. 校验会话归属并取得 userId。
2. 清理文件名（`sanitizeFileName`）。
3. 写入本地用户级 uploads 目录（`UserWorkspaceStorageService.write`）。
4. 上传到当前用户 AIO Sandbox 的 `/home/gem/uploads/{fileName}`。
5. 同名时返回 409，由前端选择替换或保留两份。
6. Office 文件触发异步预转换，便于后续预览。

### 5.2 下载

下载从 `uploads/users/{userId}/uploads/{safeFilename}` 读取。按会话所属 userId 访问用户级目录，不依赖旧的会话目录。

### 5.3 删除

删除会删除本地 `uploads/users/{userId}/uploads/{safeFilename}` 同一路径文件。

### 5.4 路径查询

`GET /api/files/path/{sessionId}` 返回本地用户上传根目录和 Sandbox 挂载路径 `/home/gem/uploads`。

## 6. 文件读写工具方法

`UserWorkspaceStorageService` 提供基础文件操作：

| 方法 | 行为 | 异常处理 |
| --- | --- | --- |
| `write(target, content)` | 创建父目录并写入字节 | 失败抛 `RuntimeException` |
| `read(source)` | 读取全部字节 | 失败抛 `RuntimeException` |
| `exists(path)` | 检查是否为普通文件 | 返回 boolean |
| `delete(path)` | 删除文件（不存在不报错） | 失败抛 `RuntimeException` |

## 7. 沙箱文件同步与恢复

### 7.1 FileSyncService

`FileSyncService` 在沙箱创建时把用户文件同步到新沙箱：

1. `syncUserWorkspace(userId, client)` 把用户上传目录和知识库目录同步到新沙箱。
2. `syncSkill(sessionId, localPath, skillId)` 把单个技能文件推送到 `/home/gem/skills/`。
3. 返回 `SyncResult`，包含 `successCount` 和 `failedPaths`。
4. 部分失败只记日志，不阻断沙箱创建。

### 7.2 KnowledgeFileMigrationService

`KnowledgeFileMigrationService` 迁移历史旧路径到用户级规范路径：

1. `migrateUser(userId)` 把旧 `rag.storage.path` 时代的文件迁移到 `uploads/users/{userId}/knowledge/`。
2. 校验文件大小后更新数据库 `storagePath`。
3. 这是历史数据迁移的兜底入口，确保旧文件也能通过用户级路径访问。

### 7.3 沙箱创建时的恢复顺序

`SandboxServiceImpl.createSandbox` 创建新沙箱后：

1. `KnowledgeFileMigrationService.migrateUser(userId)` 迁移旧知识库文件。
2. `FileSyncService.syncUserWorkspace(userId, client)` 恢复用户工作区。
3. `syncAllEnabledSkills` 同步已启用技能。

## 8. 前端工作区

`WorkspaceBrowser.js` 默认根目录是 `/home/gem`。它通过 `execute_command` 执行 `find`，构建树形结构；点击文件后调用 `FilePreviewer.preview({ source: 'workspace' })`。

当前 Chat 页面把工作区作为右侧工具 dock 的"文件"视图嵌入，不再只依赖独立浮动按钮。

## 9. 风险与排查

| 现象 | 优先检查 | 说明 |
| --- | --- | --- |
| 文件上传失败 | `sanitizeFileName`、本地目录权限、沙箱同步 | 检查文件名是否非法、`write` 是否抛异常 |
| 文件上传后沙箱看不到 | `FileSyncService` 日志、AIO endpoint | 沙箱同步部分失败只记日志 |
| 文件名含路径穿越 | `sanitizeFileName` 是否拦截 | 反斜杠和 `../` 会被清理 |
| 下载 404 | 本地 `uploads/users/{userId}/uploads/` 路径 | 文件可能被删或路径不对 |
| 知识库文件预览缺失 | `knowledge/{kbId}/` 是否同步、`sandboxSynced` | 工作区恢复部分失败只记日志 |
| 历史文件找不到 | `KnowledgeFileMigrationService` 是否迁移 | 旧路径文件需要迁移到用户级路径 |
| OSS 不可用 | `agent.storage.type` | 当前只支持 local，OSS 未启用 |
| 同名上传 409 | `FileUploadController` 同名检查 | 前端可选择替换或保留两份 |

## 10. 术语速查

本节用于 Obsidian 悬浮预览。阅读正文时，把鼠标停在带链接的术语上，可以快速看到这里的释义。

### UserWorkspaceStorageService

`UserWorkspaceStorageService` 管理用户级文件存储路径，替代了旧版全局 `rag.storage.path`。提供 `uploadFile`、`knowledgeFile`、`uploadSandboxPath`、`knowledgeSandboxPath` 等路径构造方法，以及 `write`、`read`、`exists`、`delete` 等基础文件操作。`sanitizeFileName` 处理路径穿越和非法字符。

### FileSyncService

`FileSyncService` 在沙箱创建时把用户上传目录和知识库目录同步到新沙箱，返回 `SyncResult`。部分失败只记日志，不阻断沙箱创建。还负责技能文件推送。

### KnowledgeFileMigrationService

`KnowledgeFileMigrationService` 迁移历史旧路径知识库文件到当前用户级规范路径，校验大小后更新数据库 `storagePath`。是旧数据迁移的兜底入口。

### sanitizeFileName

`sanitizeFileName` 清理文件名：反斜杠转正斜杠、取最后一段去掉路径前缀、检查空和 `.`/`..`。只做文件名清理，不做类型判断。任何扩展名都能通过。

### SyncResult

`SyncResult` 是 `FileSyncService.syncUserWorkspace` 的返回值，包含 `successCount` 和 `failedPaths`。`allSucceeded()` 判断是否全部成功，部分失败只记日志。
