# 文档摄取与切片

## 上传流程

知识库文档上传由 `RagController` 和 `KnowledgeServiceImpl` 处理：

1. 校验知识库属于当前用户。
2. 保存原始文件到用户级知识库目录。
3. 使用文档解析能力抽取文本。
4. 按配置进行切片。
5. 保存文档和切片元数据到 MySQL。
6. 调用 Embedding 服务生成向量。
7. 如果 Milvus 启用，将向量写入集合；否则由 Noop 向量服务跳过。
8. 同步知识库文件到用户 Sandbox 的 `/home/gem/knowledge/{kbId}`。

## 切片规则

默认 chunk size 是 500，overlap 是 50。切片需要保留足够上下文，同时不能过长影响检索和 rerank 成本。

## 文件迁移

`KnowledgeFileMigrationService` 负责把旧路径中的知识库文件迁移到用户级规范目录，并在校验大小后更新数据库。这个迁移会在 Sandbox 创建和工作区恢复前运行，避免旧文件路径造成预览或同步失败。

## 替换和删除

- 替换文档会重新解析、切片、Embedding 和向量写入。
- 删除文档会删除 MySQL 元数据，并调用向量服务删除对应 docId。
- 删除知识库会删除该知识库下文档和向量。
- Milvus 关闭时，向量删除由 Noop 实现记录跳过。

# 收件箱

这里用于临时放置尚未确认归属的发现。

整理规则：

1. 能从代码、配置、测试或契约确认的内容，移动到对应模块或流程页并标记 `verified`。
2. 需要运行验证或产品判断的内容，移动到 [[待确认问题]] 或 [[当前已知风险]]。
3. 不要让收件箱成为长期文档；每次知识库维护都应清理。


# 文件上传与预览

## 普通上传

1. 用户在 Chat 页面选择文件。
2. 前端调用 `POST /api/files/upload`，表单包含 file 和 sessionId。
3. `FileUploadController` 校验会话并取得 userId。
4. 文件先保存到本地用户级 uploads 目录。
5. 再上传到 AIO Sandbox `/home/gem/uploads/{fileName}`。
6. 上传成功后，用户消息附加“上传的文件”说明，Agent 系统提示会获得路径上下文。
7. Office 文件异步触发预转换。

## 重复文件

同名文件时后端返回 409。前端 DuplicateHandler 允许：

- 替换。
- 保留两份。
- 跳过。

替换走 `PUT /api/files/upload/replace`。

## 工作区浏览和预览

1. 用户打开 Chat 右侧工具 dock 的“文件”。
2. `WorkspaceBrowser` 调用 `execute_command` 执行 `find /home/gem`。
3. 前端解析输出构建文件树。
4. 点击文件后调用 `FilePreviewer.preview({ source: 'workspace' })`。
5. `FilePreviewer` 通过 `GET /api/sessions/{id}/files/preview?path=...` 拉取字节并按类型渲染。
6. 下载时调用 `GET /api/sessions/{id}/files/download?path=...`。

## 知识库预览差异

知识库文档预览除了原文，还会加载切片列表、偏移和元数据。用户可以搜索切片、展开内容、复制切片，并从切片跳回原文位置。

## 产物预览

assistant 工具产生的正式文件会进入 `chat-artifacts`。图片产物进入图库，Office/PDF/文本文件进入统一预览器。`browser_screenshot` 只有显式交付时才显示。
