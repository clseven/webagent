# 当前已知风险

## 沙箱视图代理风险

- HTTP 代理和 WebSocket 桥接已有路径/token 测试，但仍需要真实 AIO noVNC、code-server、terminal 端到端验证。
- token 只存在内存中，服务重启后旧 iframe URL 会失效，需要前端重新请求 view-url。
- 代理按用户 token 访问用户最新 endpoint，若 token 泄露，在有效期内可能访问该用户沙箱视图。
- Location 改写和 WebSocket 子协议透传需要继续覆盖更多 AIO 页面形态。
- 强制 HTTP/1.1 是为了规避 h2c 升级导致的 502，如果 AIO 升级支持 h2c 需要重新评估。

## 多租户和路径风险

- 文件、知识库、会话、应用、Sandbox 都必须按 userId 校验归属。
- 文件路径需要持续防止 `../`、绝对路径、URL 编码绕过（`sanitizeFileName` 处理）。
- docId/sessionId/appId 等直接传参接口需要系统性越权测试。
- 用户动态 MCP 默认禁止 HTTP 和私网访问是必要的 SSRF 防线，不应为了方便随意打开。

## RAG 风险

- `RAG_MILVUS_ENABLED=false` 是当前默认值，开发环境能启动，但 RAG 检索效果会退化为空向量召回。
- Query Rewrite 默认回退 DeepSeek（不是旧文档的 Ollama/qwen3），需确认 API Key 可用。
- Rerank 依赖本地 BGE 服务（`http://127.0.0.1:8002/rerank`），服务不可用时降级为向量分数排序。
- 文档异步处理没有事务包裹，失败前可能已写入切片和向量，形成脏数据残留。
- `READY` 不等于沙箱已同步：`syncToSandbox` 失败只置 `sandboxSynced=false`，状态仍为 `READY`。
- `deleteAllByUser` 不清理沙箱文件，用户注销后沙箱下会残留孤儿文件。

## MCP 风险

- shell transport 在 Sandbox 中启动 supergateway，涉及端口分配、长进程清理和协议解析，真实稳定性需要压测。
- 新增 MCP server 后工具出现有时需要下一轮 Agent loop，产品提示需保持清楚。
- 用户动态 MCP 的 SSRF 防线（禁止 HTTP、私网）不应随意打开。

## Agent 执行风险

- TodoState 是内存态，服务重启后不会保留。
- 普通工具调用当前仍以单 tool call 为主，并行主要依赖 `run_subagent`。
- 子代理后台执行需要避免最终回答早于后台结果消化。
- Hook 注入消息很强大，但新增 Hook 时要注意同步/流式两条路径一致。
- TurnPolicy 的意图判断本身消耗一次 LLM 调用，但没有单独记录 token。
- PlanAgent 规划校验两次失败走 fallback，fallback 只保留通用策略，不含工具名称。
- 大输出工具：`read_file` 已改为按行窗口分页，单条 observation 有界；但 shell/MCP/文档解析等一次性输出仍只靠 `largeOutputHook` 告警（流式路径、不截断），单条巨大结果仍可能撑爆上下文，head+tail 截断（`OutputTruncator`）是 TODO。

## 前端风险

- Chat 页面和 `style.css` 变动很大，缺少浏览器自动化测试。
- 右侧 dock 在移动端隐藏，移动端沙箱/工作区体验需要单独验证。
- 图片 lightbox、Office 转换、工作区树和 SSE 事件恢复都依赖较多运行时状态。
- SSE 断开后不可自动重连，需要用户重新发送消息。
