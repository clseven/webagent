# 面试必备内容

> 目标：面试前用这份文档快速复习 WebAgent、票联行抢票系统、实习经历和常见追问。对外简历项目名使用 **WebAgent**，仓库和知识库内部仍可能出现 `webagent-clean`。

## 0. 复习优先级

### 必须背熟

1. WebAgent 一句话定位、整体链路、最难点。
2. Agent 编排：PlanAgent、ReactAgent、Hook、TodoState 各自职责。

5. Sandbox：为什么从会话级改成用户级、同源代理解决什么问题。

### 可以现场发挥

- 前端 Chat 工作台细节。
- Office / PDF / 图片预览体验。
- TodoState 后续优化。
- 端到端测试和安全校验补强。

## 2. WebAgent 必备

### 一句话定位

WebAgent 是一个基于 Spring Boot + Vue 的多用户 AI Agent 工作台。用户通过网页会话发起复杂任务，系统用 PlanAgent 进行任务规划，用 ReactAgent 调用工具执行，并结合 RAG、MCP 动态工具、AIO Sandbox、浏览器观察、文件预览和 SSE 流式响应，完成从任务理解、计划拆解、工具执行到结果交付的闭环。

### 整体链路

```text
Vue Chat
  -> AgentController
  -> AgentServiceImpl
  -> AgentTurnContextService
  -> AgentPlannerService / PlanAgent
  -> ReactAgentFactory / ReactAgentHookService
  -> ReactAgent
  -> Tool / MCP / Sandbox / RAG
  -> SSE 事件 + assistant 消息保存
```

### 面试表达

这个项目不是简单调用大模型，而是把模型、工具、知识库、沙箱、文件系统和前端实时状态做成一个可执行闭环。模型不是直接操作系统资源，而是通过统一 Tool 接口发起工具调用；工具执行后返回 observation，再由 ReactAgent 决定下一步。前端通过 SSE 展示 plan、thinking、tool_call、observation 和 answer，让用户能看到任务执行过程。

### 为什么要拆 AgentServiceImpl

如果 `AgentServiceImpl` 同时负责上下文准备、规划、工具装配、MCP、RAG、Hook、模型调用、消息保存和 SSE 转发，会变成一个很难维护的巨型服务。拆分后：

- `AgentTurnContextService` 负责准备历史消息、文件上下文、知识库上下文和工具上下文。
- `AgentPlannerService` 负责调用 PlanAgent 做任务规划。
- `ReactAgentFactory` 负责创建同步或流式执行器。
- `ReactAgentHookService` 负责注册 Hook，例如图片观察、最终回答门禁（TodoState + 证据自检）和子代理。
- `AgentServiceImpl` 保留入口编排、会话校验、消息保存和 token 记录。

这样同步调用和 SSE 流式调用能复用同一套上下文和执行器构造逻辑。

## 3. Agent 编排必备

### PlanAgent

职责：任务前规划。它不直接执行工具，而是根据用户输入、历史上下文、应用配置和可用工具，生成目标、执行策略、成功信号和初始步骤。

面试说法：

> 我把 PlanAgent 定位成“任务规划器”，它负责告诉执行阶段要做什么、什么算完成。这样可以避免 ReactAgent 一上来就盲目调用工具，也方便前端先展示 plan 事件。

### ReactAgent

职责：工具执行循环。它根据模型响应决定调用哪个工具，工具返回 observation 后再继续推理，直到满足最终回答条件。

面试说法：

> ReactAgent 更像执行器，它维护多轮 tool call -> observation -> model reasoning 的循环。工具本身不直接生成最终答案，而是把结果返回给模型，由模型结合上下文继续判断下一步。

### Hook

职责：处理横切逻辑，避免把所有边界行为塞进 ReactAgent 主循环。

典型 Hook：

- PreToolUseHook：工具调用前检查。
- PostToolUseHook：工具调用后补充行为，例如图片观察。
- StopHook：最终回答前检查，例如 TodoState 是否允许结束。

面试说法：

> 我们没有把图片观察、最终回答门禁（TodoState + 证据自检）这些逻辑直接写死在工具或主循环里，而是通过 Hook 注入。这样新增一个横切能力时，不需要重写 ReactAgent 的核心循环。

### TodoState

职责：执行中的任务状态账本，不是调度器。

它记录：

- 当前任务是否完成。
- 证据是什么。
- 是否存在阻塞。
- 是否允许最终回答。

面试说法：

> TodoState 不是用来调度工具的，而是用来防止多步任务还没做完就提前总结。调度仍由 ReactAgent 和工具循环负责，TodoState 只记录可验证状态和最终回答门禁。

## 4. MCP 必备

### MCP 解决什么问题

传统工具如果都写死在后端代码里，新增工具需要改代码、发版，扩展成本高。MCP 的价值是把外部工具以统一协议暴露给 Agent，WebAgent 可以动态发现并接入这些工具。

### WebAgent 怎么接 MCP

核心链路：

```text
MCP Server
  -> McpClientToolProvider 拉取工具列表
  -> RealMcpTool 包装成统一 Tool
  -> ReactAgent 工具集合
  -> 模型选择工具并发起调用
```

面试说法：

> MCP 工具在系统里不是特殊通道。我们会把 MCP Server 返回的工具包装成 RealMcpTool，再进入和本地工具一样的 Tool 调用流程。这样 ReactAgent 不需要关心工具来自本地还是 MCP，只要按统一定义调用即可。

### MCP 工具命名冲突

可能出现不同 MCP Server 暴露同名工具的问题。处理思路：

- 根据 serverId 或命名规则生成唯一工具名。
- 保存 originalName 和 serverId。
- 对模型暴露唯一 name，对真实调用保留原始工具名。

### MCP 安全边界

必须能说清楚：

- 用户级 MCP 配置要按 userId 隔离。
- 动态 MCP 默认限制私网地址，避免 SSRF。
- shell transport 只能在 Sandbox 内运行，不能在宿主机直接执行。
- 工具参数需要校验，不能完全信任模型生成的参数。

面试追问：为什么 MCP 有安全风险？

答：

> 因为 MCP 本质上让模型有机会调用外部工具。如果允许任意 URL 或 shell 命令，就可能出现 SSRF、越权访问或命令执行风险。所以要限制 transport、校验地址、隔离用户配置，并把高风险执行放进 Sandbox。

## 5. RAG 必备

### 上传链路

```text
文件上传
  -> 保存原文件
  -> 文档解析
  -> 文本切片
  -> Embedding
  -> 向量写入 Milvus / Noop
  -> 元数据写入 MySQL
```

### 检索链路

```text
用户问题
  -> Query Rewrite
  -> 多 query Embedding
  -> 向量召回
  -> 候选切片去重
  -> BGE Rerank
  -> topK 上下文注入 Agent
```

### Query Rewrite 的价值

用户问题经常口语化、指代不清或上下文依赖强。Query Rewrite 根据当前问题和历史上下文生成更适合检索的查询语句，提高语义召回效果。

面试说法：

> Query Rewrite 不负责回答用户，它只负责生成更适合检索的 query。比如用户问“刚才那个功能怎么做”，原始 query 很难检索，rewrite 会结合上下文补全真实检索关键词。

### Rerank 的价值

向量召回更关注语义相似度，但 topK 里可能有噪声。Rerank 根据原始问题重新排序候选切片，让最终注入给模型的上下文更准确。

### Milvus 默认关闭怎么解释

面试说法：

> 当前本地开发默认可以关闭 Milvus，这样系统容易启动，但不代表完整 RAG 演示状态。真正演示 RAG 效果时，需要打开 Milvus 并确认集合里已有向量数据。

## 6. Sandbox 必备

### 为什么从会话级 Sandbox 改成用户级 Sandbox

会话级 Sandbox 的问题：

- 多个会话间文件和环境不能复用。
- 同一个用户跨会话工作区割裂。
- endpoint 变更后恢复复杂。

用户级 Sandbox 的价值：

- 一个用户复用同一个 Sandbox。
- 多个会话共享文件、浏览器和工作区状态。
- 通过 userId 查最新 sandboxId 和 endpoint，便于恢复。

### 为什么前端不直接访问 AIO endpoint

原因：

- 跨域问题。
- endpoint 可能变化。
- 不能把内部 endpoint 暴露给前端。
- 需要按用户权限校验。

解决方案：

```text
前端 iframe / WebSocket
  -> /sandbox-view/{token}/...
  -> 后端校验 token
  -> 根据 userId 查最新 AIO endpoint
  -> 代理 HTTP / WebSocket 到 Sandbox
```

面试说法：

> token 里不直接保存 endpoint，只保存能定位用户的信息。每次代理请求都根据 userId 查最新 endpoint，这样 Sandbox 重建后 endpoint 变化也能恢复。

## 7. SSE 与前端工作台必备

### 为什么不用 EventSource

可以这样答：

> EventSource 只支持 GET，扩展请求体、鉴权和复杂参数不方便。前端使用 fetch + ReadableStream 能发送 POST 请求，也能更灵活地解析服务端返回的 SSE 事件。

### SSE 展示哪些事件

- `plan`：任务规划。
- `thinking`：模型思考或阶段提示。
- `tool_call`：工具调用开始。
- `observation`：工具返回结果。
- `answer`：最终回答。

### 前端工作台重点

- 右侧工具 dock 承载 Sandbox 和工作区。
- 文件产物从工具事件中提取并去重。
- 图片 lightbox 支持同一消息内多图预览。
- Office / PDF / 文本统一预览。
- `deliver_to_user` 防止过程截图污染最终交付图库。

## 8. WebAgent 高频问答

### Q1：这个项目最难的点是什么？

难点不是单次调用模型，而是让模型、工具、沙箱、文件、RAG 和前端实时状态形成闭环。模型调用工具后返回 observation，Hook 可能注入图片观察，TodoState 可能阻止提前最终回答，前端还要把这些事件恢复成可读的行动流。这些模块既要解耦，又要在一次任务里串起来。

### Q2：怎么保证多用户隔离？

业务数据都带 userId；会话、知识库、上传文件、Sandbox 和 MCP 配置都按当前用户校验。Sandbox 从会话级改成用户级后，通过 `UserSandboxEntity` 管理 sandboxId 和 endpoint；前端只拿短期 token 访问同源代理，不能直接访问内部 endpoint。

### Q3：为什么要做 MCP？

如果所有工具都写死在后端，新增工具需要改代码和发版。MCP 让外部工具可以用统一协议暴露给 Agent，系统通过 McpClientToolProvider 动态拉取工具，再包装成 RealMcpTool 进入统一调用流程。这样本地工具和外部 MCP 工具都能被 ReactAgent 以同一种方式调用。

### Q4：RAG 如何避免幻觉？

RAG 本身不能完全消灭幻觉，但可以减少模型凭空回答。系统通过 Query Rewrite 提升召回，通过 Rerank 降低噪声，再把 topK 切片作为上下文注入 Agent。同时 Prompt 要约束模型优先依据资料回答，不足时说明资料不足。

### Q5：Hook 和工具有什么区别？

工具是模型主动调用的能力，比如浏览器截图、文件读取、知识库搜索。Hook 是系统在工具调用前后或最终回答前自动触发的横切逻辑，比如图片观察、最终回答门禁（TodoState 前置 + 证据自检）。Hook 不需要模型显式选择。

### Q6：如果让你继续优化 WebAgent，你会做什么？

优先补端到端测试，覆盖 SSE 流式事件、真实 AIO Sandbox 视图、MCP 工具生命周期和多用户越权校验。然后考虑 TodoState 持久化、普通多 tool call 支持、移动端工具 dock 体验和 MCP 工具权限分级。

## 9. 票联行高并发抢票系统必备

### 一句话定位

票联行是面向演出票务场景的高并发购票系统，支持注册登录、节目列表 / 详情、座位查询、订单生成、支付、订单查询，并围绕抢票链路做分库分表、缓存防护、库存扣减、订单超时取消和智能服务模块。

### 分库分表怎么讲

问题：

- 用户量和订单量增长后，单表查询和写入压力变大。
- 用户可能通过手机号、邮箱、用户 ID、订单号等多个维度查询。

方案：

- 设计手机号 / 邮箱路由表，建立用户标识和用户 ID 的映射关系。
- 基于用户 ID 基因生成订单编号，让订单号里带有可路由信息。
- 用户维度查询走用户 ID，订单维度查询可根据订单号定位分片。

面试说法：

> 分库分表后最麻烦的是路由问题。我的处理思路是先把手机号、邮箱映射到用户 ID，再基于用户 ID 基因生成订单号。这样用户登录查询和订单号查询都能定位到对应分片，避免全库扫描。

### 缓存击穿与穿透怎么讲

击穿：

- 热点 key 失效，大量请求同时打到数据库。
- 方案：读写锁、双重检测、多级缓存，控制只有少量线程回源加载。

穿透：

- 查询不存在的数据，缓存中没有，数据库也没有。
- 方案：布隆过滤器拦截不存在用户或无效请求。

### 订单超时处理怎么讲

业务要求：

- 订单创建后 15 分钟未支付要自动取消。
- 座位库存要及时释放。

方案：

- 使用 Redisson 延迟队列保存超时任务。
- 任务分片 + 线程池消费，避免单线程处理慢。
- 到期后检查订单状态，未支付才取消并释放库存。

注意点：

- 取消前必须再次查订单状态，防止用户刚支付成功又被取消。
- 释放库存和订单状态变更要考虑幂等。

### Redis + Lua 扣库存怎么讲

为什么用 Redis：

- 抢票瞬时并发高，直接打 MySQL 压力大。
- 余票读写频繁，适合放 Redis。

为什么用 Lua：

- 查询库存和扣减库存必须原子执行。
- Lua 在 Redis 内单线程执行，可以避免并发下超卖。

为什么用 Kafka：

- Redis 扣减成功后，数据库同步可以异步化。
- Kafka 用来削峰，把扣减结果异步写回 MySQL。

### Spring AI 智能服务模块怎么讲

定位：

> 它不是替代核心购票链路，而是在购票系统上扩展智能客服能力，让用户通过自然语言查询节目、订单状态和购票规则。

技术方案：

- 使用 Spring AI 接入大模型。
- 封装节目搜索、订单查询、购票规则咨询等 Tool Calling 工具。
- 工具内部仍调用后端已有 Service，不让模型直接访问数据库。
- Prompt 约束模型优先调用工具或依据规则回答，减少幻觉。

面试说法：

> 我没有让大模型直接操作数据库，而是把节目搜索、订单查询、规则咨询封装成工具。模型只负责理解用户意图和选择工具，真正的数据查询仍走后端 Service，这样可控性和安全性更好。

## 10. 实习经历必备

### LLM 调用治理

可讲点：

- 不同模型接口和响应格式不一致。
- 统一调用入口和响应结构。
- 增加 Token 用量统计和日志记录。
- 为后续模型扩展留接口。

面试说法：

> 我做的是调用链路治理，不只是接一个模型接口。重点是把不同模型的请求、响应、错误处理和 token 统计统一起来，后续接新模型时不需要改业务层代码。

### Query Rewrite

可讲点：

- 用户问题口语化，直接检索效果不稳定。
- 根据用户问题和上下文改写检索 Query。
- 目标是提升召回，不是生成答案。

### ASR/TTS

可讲点：

- 腾讯云 ASR：语音转文字。
- 腾讯云 TTS：文字转语音。
- 后端封装接口，配合前端联调实时语音识别和播报。

### OCR POC

可讲点：

- POC 重点是验证客户侧效果。
- 将 OCR 结果转换为 JSON / HTML 结构化内容。
- 方便前端展示和客户演示。

### 并发日志顺序错乱

可讲点：

- 异步执行链路里节点日志和大模型日志可能并发写入。
- 排查日志写入顺序、事件时间、异步任务回调。
- 目标是让流程执行记录和真实执行顺序一致。

## 11. 面试中不要这么说

不要说：

- “我就是调用了一下大模型接口。”
- “MCP 就是一个插件。”
- “RAG 可以解决幻觉。”
- “Redis + Lua 一定不会有并发问题。”
- “Spring AI / LangChain 都差不多。”
- “这个项目是我学习用的。”

更好的说法：

- “我做的是大模型调用链路治理和工具执行闭环。”
