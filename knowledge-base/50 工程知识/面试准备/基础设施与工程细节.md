# 基础设施与工程细节

> 用途：这份集中"硬 Java 工程点"——网络协议、WebSocket、代理、认证、并发。这些是简历上看不出、但一深挖就见功底的地方，也是资深面试官最爱切入的。
> 每个点按「现象/需求 → 怎么解决 → 深挖」组织。

---

## 一、为什么强制 HTTP/1.1（王牌深挖点）

### 现象

后端用 JDK HttpClient 访问 AIO 沙箱的 endpoint（健康检查、视图代理），如果不强制 HTTP/1.1，会 502 Bad Gateway，健康检查误判沙箱不健康。

### 根因

> JDK HttpClient 默认走 HTTP/2。在**明文（非 TLS）连接**上，它会尝试 h2c 升级——发一个带 `Upgrade: h2c` 和 `HTTP2-Settings` 头的请求。但沙箱容器里的 nginx / code-server 后端不支持 h2c，收到这些升级头就返回 502。所以我在所有访问沙箱的 JDK HttpClient 上显式 `.version(HTTP_1_1)`，禁掉 h2c 升级。

### 深挖延伸（答出这些=懂网络）

- **h2c vs h2**：h2c 是明文 HTTP/2，靠 Upgrade 头协商；h2（TLS 上的 HTTP/2）靠 TLS 的 ALPN 协商，不会有这个问题。
- **为什么 WebClient（Reactor Netty）主通道不用显式设**：Reactor Netty 默认就是 HTTP/1.1，不会主动 h2c 升级；问题只出在 JDK HttpClient。
- 代码位置：`AioHttpClient` 的健康检查专用 client、`SandboxViewProxyController` 的代理 client 都显式 HTTP_1_1。

（这条已沉淀为记忆 aio-sandbox-requires-http1_1）

## 二、AIO 客户端架构（门面 + 传输 + 领域 API）

### 设计

三层：
- **AioClient**：聚合门面。一个 baseUrl 建一个共享 AioHttpClient，组装 6 个领域 API（sandbox/shell/file/browser/node/utility），对外用 `shell()`/`files()`/`browser()` 访问。
- **AioHttpClient**：唯一 HTTP 传输层。封装 WebClient、超时（120s）、maxInMemorySize（16MB）、统一信封解析 `{success, message, data}`（success=false 抛 AioApiException）、二进制、multipart。
- **领域 API**：薄封装，只拼路径 + 调传输层。

### 亮点

**健康检查用独立的安静 client**（getTextQuietly）：健康探针预期会失败（沙箱没起来时连接拒绝），如果用主 WebClient 会刷一堆 WARN 日志。所以专门另起一个 JDK HttpClient 做健康检查，避免污染日志。

### 深挖坑

- **AioClient 每次 new**：SandboxClientFactory 每次 `new AioClient("http://"+endpoint)`，即每次新建 WebClient/连接池，没缓存复用 → 性能优化切入点。
- AioShellApi 有个可变字段 shellSessionId（默认复用会话），多线程复用同一 AioClient 时有竞态。

## 三、沙箱视图同源代理（云端部署的关键）

### 为什么需要

前端要在 iframe 里显示沙箱的 noVNC、terminal、code-server。但**不能让浏览器直接连 AIO endpoint**：
- endpoint 常是 `127.0.0.1:随机端口`，云端部署时浏览器会把 127.0.0.1 解析到**用户自己的电脑**，根本连不上。
- 跨域、endpoint 会变、不能暴露内部地址、要按用户鉴权。

### 方案

浏览器只访问同源 `/sandbox-view/{token}/...`，后端代理到真实沙箱。三个组件：

**1. Token 服务（SandboxViewTokenService）**
- issue：SecureRandom 24 字节 → Base64 URL-safe token，存内存 Map，TTL 30 分钟。
- **token 只绑 userId，不固定 endpoint**——每次请求重新按 userId 查最新 endpoint，所以沙箱重建/换端口后旧 token 仍可用。
- 深挖坑：纯内存、无持久化、惰性过期（只在 resolve 时清理）、无撤销机制、多实例不共享 → 引出 Redis 化。

**2. HTTP 代理（SandboxViewProxyController）**
- 用请求头 `headers="!Sec-WebSocket-Key"` 排除 WebSocket 握手，把普通 HTTP 和 WS 分流——**用请求头做路由分流**的巧写法。
- 强制 HTTP_1_1 + Redirect.NEVER：不跟随重定向，而是把上游返回的 Location 头**改写**回同源 `/sandbox-view/{token}/...`，避免浏览器跳到 127.0.0.1。
- 过滤 hop-by-hop 头，注入 X-Forwarded-Host/Proto/Prefix。

**3. WebSocket 代理（SandboxViewWebSocketProxyHandler）**
- 握手时解析 token→userId→endpoint，连上游建立 Bridge，downstream↔upstream 双向转发。

### WebSocket 缓冲区调优（第二个王牌深挖点）

> Spring 默认 WebSocket text buffer 只有 8KB。但 code-server 的管理通道单条消息能到 9KB 以上，超了会触发 **1009 Message Too Big** 直接断连。所以我把两端的 buffer 都调大到 64KB/256KB——浏览器侧在 WebSocketConfig 配 ServletServerContainerFactoryBean，上游侧通过 ContainerProvider 拿独立容器调大。

其它 WS 细节：
- **copyMessage**：跨两个 session 转发前必须复制消息，否则同一 buffer 复用会状态错乱。
- **synchronized(session) 写**：WebSocket 规范不允许并发写同一 session，所以发送加锁保证单线程写。

## 四、认证与多用户隔离

### 两套独立认证体系（重要）

> 我这个系统有两套认证，走不同路径：
> 1. **业务 API**（/api/*）：AuthFilter 拦截，要求 Bearer token，查库拿 userId 放进 UserContext。
> 2. **沙箱视图**（/sandbox-view/*）：不走 AuthFilter（不是 /api/），完全靠 view token 鉴权。
> 分开是因为视图代理是给 iframe/WebSocket 用的，没法带 Authorization 头，只能靠 URL 里的短期 token。

### UserContext = ThreadLocal

AuthFilter 在请求进来时 `UserContext.setCurrentUserId(id)`，**finally 里 clear()**。所有业务按 UserContext 拿当前用户实现隔离。

### 深挖坑（能主动讲=加分）

- **必须 finally clear**：线程池复用线程，不清理会串号（A 的请求用了 B 残留的 userId）。代码做了。
- **ThreadLocal 在流式/异步不传播**：这个项目大量用 SSE 流式（chatStream），ThreadLocal 不会自动传到 Reactor 线程/子线程。所以异步链路里我不依赖 ThreadLocal，而是用 sessionId 查库拿 userId（前面 createSandbox、MCP 都是这么做的）。这是这套设计最大的隐患点，值得主动提。
- **token 是 UUID，无过期无 JWT**：登录用 BCrypt 校验密码，token 用 UUID，每次登录重新签发（踢掉旧会话）。可优化成 JWT + 过期 + 刷新。

## 五、Office 文件预览

### 实现

沙箱内跑 `soffice --headless --convert-to pdf` 把 Office 文件转 PDF 给前端预览。

### 几个考究的设计

- **缓存键含源文件 sha256**：缓存文件名是 `{docId}-{sourceHash}.pdf`，源文件一变 hash 变、缓存自动失效。
- **soffice 独立 profile**：用 `-env:UserInstallation` 给每次转换独立 profile，避免并发转换互相干扰。
- **路径白名单防注入**：强制路径以 `/home/gem/` 开头、拒绝 `..` 和空字符；所有 shell 参数用单引号转义。

### 深挖坑

LibreOffice 转换重、慢、可能超时（靠 timeout 命令兜底）；多实例共享沙箱文件系统缓存。

## 六、面试深挖点速查（按"王牌程度"排序）

| 点 | 一句话 | 深挖到哪算到位 |
| --- | --- | --- |
| **HTTP/1.1 强制** | h2c 明文升级触发 nginx 502 | 能讲 h2c vs h2、ALPN |
| **WS 8KB→64KB** | 1009 Message Too Big | 两端都调、copyMessage、单线程写 |
| **同源代理** | 云端 127.0.0.1 解析到用户本机 | 不跟随重定向 + Location 改写 |
| **view token** | 只绑 userId 不绑 endpoint | 内存态缺陷 → Redis 化 |
| **ThreadLocal 流式不传播** | 异步用 sessionId 查库替代 | filter finally clear 的必要性 |
| **两套认证** | /api Bearer vs /sandbox-view token | 为什么分开 |
| **AioClient 无连接池复用** | 每次 new | 性能优化切入点 |
| **Office 缓存 sha256 + profile 隔离** | 源变缓存失效、并发隔离 | 路径白名单防注入 |
