# 第一部分：项目亮点（主动抛，引导深挖）

## 3 分钟技术自述（开场用）

> WebAgent 这个项目我觉得最有技术含量的是三块，我可以展开讲：
>
> **第一是 Agent 执行引擎**。我没有用现成框架，自己实现了 PlanAgent 规划 + ReactAgent 执行的循环。里面几个我比较满意的设计：一是 Hook 机制，把图片观察、最终回答门禁这些横切逻辑从主循环里解耦出来；二是工具的并发调度，按副作用把工具分成可并发的读和必须串行的写；三是双层流式，既推步骤级事件也推 token 级事件。
>
> **第二是上下文工程**。多轮 + 多工具调用的对话很快会撑爆上下文，我做了超阈值自动压缩——把旧消息交给模型摘要、保留最近部分，切分点还要对齐工具调用边界。同时配合 prompt caching 组织请求前缀，省 token 也降延迟。
>
> **第三是沙箱与同源代理**。每个用户一个长期沙箱，前端要访问沙箱里的 noVNC、terminal、code-server，但不能直接连内部地址，我用短期 token + 后端同源代理解决了跨域、鉴权和 endpoint 变化的问题，HTTP 和 WebSocket 都代理了。
>
> 这三块背后其实是同一个主题：让模型、工具、沙箱、知识库和前端实时状态形成一个可执行、可观察的闭环，而不是简单调一次模型。

**（抛完停住，让面试官挑一个问。每一块在第二部分都有深挖版。）**

## 一句话难点（被问"最难的点"时用）

> 最难的不是单次调模型，而是让模型、工具、沙箱、文件、RAG 和前端实时状态形成闭环。模型调完工具返回 observation，Hook 可能注入图片观察，TodoState 可能拦截提前收尾，前端还要把这一串事件恢复成可读的行动流。这些模块既要解耦、又要在一次任务里精确串起来，还要处理各种边界——上下文超长、工具超时、迭代上限。难的是"编排"和"边界"，不是"调用"。

# 第二部分：核心模块深挖

## 模块 1：ReAct 执行循环

### 一轮循环具体怎么转？

> 一轮是这样：
> 1. 调 LLM 之前先做上下文压缩检查（超阈值就压）。
> 2. 调 LLM，让它看当前所有历史决定下一步。
> 3. 如果返回的是工具调用——收集本轮**全部** tool_call，交给 ToolScheduler 调度执行，结果按原顺序追加回消息列表，继续下一轮。
> 4. 如果返回的是最终答案——先过 StopHook 自检，放行就结束返回，不放行就强制继续。
> 5. 有个 MAX_ITERATIONS 上限（25 轮），到顶强制停止返回提示，防止死循环烧钱。
> 核心是每一轮的思考、动作、观察都追加进 messages，模型能看到完整历史，实现自我纠错。

### 多个工具调用怎么调度？（ToolScheduler + 副作用分类）

> 模型一轮可能返回多个工具调用。我按**副作用类型**决定能不能并发，这是我比较得意的一个设计：
> - **READ**（纯读，如 web_search、read_file、grep）：互不干扰，可以并发跑。
> - **WRITE**（改文件/沙箱，如 write_file）：串行，且和 READ 互斥。
> - **EXCLUSIVE**（改环境/全局副作用，如 execute_command、git）：完全串行独占。
>
> **关键设计是"默认保守"**：所有工具默认按最严的 EXCLUSIVE 串行对待，只有我确认无副作用的工具才**显式**标成 READ。这样漏标一个工具，最坏就是失去并发收益（慢一点），但绝不会把有副作用的工具误当并发跑导致数据竞争。宁可慢、不可错——这是并发安全的保守原则。
> 单个工具调用有快捷路径，不走调度器，避免开销。

### MAX_ITERATIONS 上限会不会误伤正常的长任务？

> 有可能，25 轮是个权衡值。它防的是两种失控：一种是模型陷在重复调工具的死循环，一种是过度谨慎反复查证不收尾。另一端由 StopHook 的收敛机制配合——两道防线一起才稳。真遇到确实需要更多轮的复杂任务，现在是返回提示让用户简化，理想是保留中间产物、可续跑，这是可优化点。

## 模块 2：Hook 机制（横切逻辑解耦）

### 为什么要有 Hook？和工具的区别？

> 工具是模型**主动选**的能力（截图、读文件、搜知识库）。Hook 是系统在特定时机**自动触发**的横切逻辑，模型不感知。如果把图片观察、最终回答门禁这些都写死进主循环，ReactAgent 会越来越臃肿，加一个横切能力就要动核心循环。Hook 让这些能力可插拔。

### 三种 Hook 的触发时机和返回语义？

> 三种 Hook，返回值语义我设计得很明确：
> - **PreToolUseHook**（工具执行前）：返回 null 放行；返回非 null 字符串就**阻止执行**，原因作为工具结果返回给模型。用于权限校验、危险操作拦截。
> - **PostToolUseHook**（工具执行后）：返回 null 无额外动作；返回非 null 的 ChatMessage 就**追加进历史**，下一轮模型可见。典型用途是图片注入——工具截图后，这个 Hook 把图片数据转成模型能看的消息。
> - **StopHook**（模型想收尾前）：返回 null 允许退出；返回非 null 字符串就**强制继续**，作为新的 user 消息注入。用于最终回答门禁——任务没真做完不许收尾。

### StopHook 的最终回答门禁怎么防"没做完就总结"，又怎么防"死循环查"？

> 这是个双向防线，我觉得设计得比较巧：
> - **一端防"过早收尾"**：StopHook（FinalTodoGuardHook）检查 TodoState，任务没完成、证据不足就不放行，强制模型继续。
> - **另一端防"过度谨慎无限查"**：StopHook 有个 finalizeAttempt 计数——模型每次准备收尾就 +1，真去调了工具就清零。如果模型连续多次想收尾却不补充查证，就强制放行，避免它陷在"我再查查"的死循环。
> - **状态归属很讲究**：计数由 ReactAgent（执行器）持有并传给 Hook，Hook 只读不持有状态。这样 Hook 是无状态的、可复用的，状态集中在执行器一处管理。

## 模块 3：上下文压缩（详见边界文档，这里讲机制）

### 压缩机制一句话

> 每次调 LLM 前估算消息总量（字符数粗估，只是触发开关），超过约 2.4 万就把最旧的一批消息交给 LLM 生成摘要，替换原文，保留最近约 40%。摘要作为普通 user 消息放回，不塞进 system prompt。

### 最关键的坑——切分点对齐

> 最容易踩的坑是切分点：如果切在一组 tool_call 和它对应的 tool_result 中间，把它们切散，模型下一轮会直接报协议错误（有 tool_call 没有对应结果）。所以我专门做了切分点对齐到工具调用边界。这个坑是我实际跑出来才发现并修的。

（token 估算为什么用粗估、真实 usage 怎么用、prompt caching 怎么配合——见 [[边界与异常处理问答]] 第 1 节。）

## 模块 4：SSE 双层流式

### "双层"是什么意思？

> 两层事件：
> - **外层：步骤级事件**——plan（规划）、thinking（思考开始）、tool_call（工具调用）、observation（工具结果）、answer（最终答案）。让用户看到 Agent 的行动流。
> - **内层：token 级事件**——LLM 输出的每个 token 实时往外推。让用户看到文字一个个蹦出来，不是等一整段。
> 两层叠加，用户既看得到"它在做什么"，也看得到"它在说什么"。后端用 Flux 实现。

### 前端为什么用 fetch + ReadableStream 而不是 EventSource？

> 浏览器原生 EventSource 只支持 GET，带不了请求体和复杂鉴权。我用 fetch + ReadableStream 手动解析 SSE，能发 POST、带完整对话参数和 token。代价是要自己处理流的解析和拼接。

### 用户中途停止怎么实现？

> 通过 Flux.create 的 emitter.isCancelled() 检测客户端断开，在循环开始时检查，实现即时中断。用户关掉或点停止，连接断开，后端下一轮循环开始时就停了。

## 模块 5：沙箱同源代理

### 前端为什么不能直接访问 AIO 沙箱？

> 四个原因：跨域；endpoint 可能变（沙箱重建后变）；内部 endpoint 不能暴露给前端；要按用户权限校验。

### 整条代理链路？

> 前端 iframe/WebSocket 访问 `/sandbox-view/{token}/...` → 后端校验 token → 按 userId 查该用户最新的 AIO endpoint → 代理 HTTP 或 WebSocket 到真正的沙箱。
> 关键设计：**token 里不存 endpoint，只存能定位用户的信息**。每次请求都实时按 userId 查最新 endpoint，所以沙箱重建、endpoint 变了也能自动恢复，不用重新签 token（除非 token 本身过期）。

### 边界（会被追问的）

> token 短期有效、只存内存、重启失效；有效期内泄露有风险，可加绑定 UA/nonce 增强。HTTP 代理要做 Location 改写，WebSocket 要透传子协议。强制 HTTP/1.1 是为了规避 h2c 升级导致的 502。（详见 [[当前已知风险]]）

## 模块 6：轮次策略（TurnPolicy，加分细节）

### 不是所有对话都要全套能力

> 我做了个 TurnPolicy——先判断这轮的意图。如果是纯闲聊（SOCIAL），就跳过工具、工作区、知识库和 StopHook，直接轻量对话，省 token 省延迟；如果是任务（TASK）或模糊（AMBIGUOUS），保持全能力。
> 这样"你好"这种不会触发一整套规划+工具装配的重流程。代价是意图判断本身消耗一次 LLM 调用——这也是我知道要优化的点（这次调用的 token 目前没单独统计，简单意图其实可以用规则先兜）。
