# 八股结合项目

> 用途：面试官最爱把八股挂到你项目上问——"你项目里线程池怎么配的""ThreadLocal 会内存泄漏吗""Spring 事务什么时候失效"。纯背八股不如"八股 + 我项目里的真实例子"，后者证明你真的用过、踩过。
> 每条：八股要点 → 我项目里的落点 → 可能的追问。

---

## 一、Spring：@Async 自调用失效（王牌）

**八股**：@Async / @Transactional 靠 AOP 代理实现。同一个类内部方法自调用（this.method()）不经过代理，注解直接失效。

**我项目里**：
> 知识库文档处理（解析+切片+向量化）很耗时，我用 @Async 异步化不阻塞上传接口。一开始想放在 KnowledgeServiceImpl 里，但发现 @Async 没生效、接口还是卡几十秒。原因就是同类自调用不走代理。我把处理链路拆成独立的 Bean（KnowledgeDocumentProcessor），从外部注入调用，@Async 才真正生效。

**追问：怎么确认它真异步了？** → 看日志线程名（异步走 knowledgeTaskExecutor 线程池，不是 http-nio 请求线程）；接口立即返回、状态是 PROCESSING。

**追问：@Transactional 还有哪些失效场景？** → 自调用、方法非 public、异常被 catch 没抛、rollbackFor 没配对 checked 异常、非 Spring 管理的对象调用。

## 二、Spring：条件化 Bean（@ConditionalOnProperty）

**八股**：Spring 可以根据配置/环境决定装哪个 Bean，实现同一接口多实现的切换。

**我项目里**：
> Milvus 向量库默认关闭（方便本地启动）。我用 @ConditionalOnProperty 让两个实现二选一：开启时装真实的 VectorStoreServiceImpl，关闭时装 NoopVectorStoreServiceImpl（所有操作空转、检索返回空）。好处是降级不用改上层代码，也不用到处写 if，任意配置下都恰好有一个 Bean 可注入。

**追问：为什么不用 @Profile？** → Profile 是环境维度（dev/prod），这里是功能开关维度（milvus 开不开），用 @ConditionalOnProperty 更贴切，可以细到单个配置项。

---

## 三、并发：ThreadLocal（王牌）

**八股**：ThreadLocal 线程隔离；线程池复用线程时不清理会串数据 + 内存泄漏（key 是弱引用、value 强引用，线程不死 value 不回收）。

**我项目里**：
> 我用 ThreadLocal（UserContext）存当前请求的 userId，实现多用户隔离。两个关键处理：
> 一是 AuthFilter 在 finally 里 clear()——因为 Tomcat 线程池复用线程，不清理下个请求会读到上个用户的 userId，直接串号。
> 二是我发现 ThreadLocal 在异步/流式场景不传播——项目大量用 SSE 流式，Reactor 线程和子线程拿不到主线程的 ThreadLocal。所以在异步链路（建沙箱、MCP）里我不依赖 ThreadLocal，而是用 sessionId 查库拿 userId。

**追问：为什么不用 InheritableThreadLocal / TransmittableThreadLocal？** → InheritableThreadLocal 只在创建子线程时传一次，线程池复用场景不行；阿里的 TTL 能解决线程池传播，但我这里更简单直接——异步任务本来就有 sessionId，查库拿 userId 更可靠、没有隐式依赖。

---

## 四、并发：线程池 / CompletableFuture（王牌，也是我的坑）

**八股**：CompletableFuture.supplyAsync 不传 executor 默认用 ForkJoinPool.commonPool；commonPool 线程数少（CPU-1）、且是全 JVM 共享，跑阻塞任务会打满、影响其它并行任务。orTimeout 只让 future 超时，不中断底层线程。

**我项目里（诚实讲坑）**：
> 几个地方用了 CompletableFuture：RAG 并行向量召回、子代理执行。这里我知道两个待改进点：
> 一是用了默认 commonPool，而这些任务是阻塞 IO（等 LLM、等 Milvus），严格讲应该用独立线程池，避免占满公共池。
> 二是子代理超时用 orTimeout，它只让主流程不等了，**底层跑 LLM 的线程并没真正中断**，还在后台烧 token。正确做法是传取消信号 + 用可中断的执行器。

**追问：@Async 的线程池怎么配的？** → 文档处理有专门的 knowledgeTaskExecutor（自定义 ThreadPoolTaskExecutor，配了核心/最大线程、队列、拒绝策略），这个是对的；问题在那几个手写的 CompletableFuture 上。

## 五、并发安全：保守设计（能讲思路）

**八股**：并发下共享状态要么加锁、要么避免共享；判断能否并行要看有没有副作用。

**我项目里（ToolSideEffect）**：
> 模型一轮可能返回多个工具调用，我按副作用决定能否并发：READ（纯读）可并发，WRITE/EXCLUSIVE（改文件/改环境）串行。
> 关键是**默认保守**：所有工具默认按最严的 EXCLUSIVE 串行，只有我确认无副作用的才显式标 READ。这样漏标一个工具最坏就是慢一点（失去并发），但绝不会把有副作用的当并发跑导致数据竞争。宁可慢、不可错。

## 六、事务：跨库一致性（我的坑，诚实讲）

**八股**：@Transactional 只管单个数据源的关系型数据库事务，管不了外部系统（MQ、ES、向量库、远程调用）。跨系统要靠最终一致性/补偿。

**我项目里**：
> RAG 写入要同时写 MySQL（切片元数据）和 Milvus（向量）。这俩是不同的存储，@Transactional 管不了 Milvus。我现在的实现是先写 MySQL 再写 Milvus，如果 Milvus 失败 MySQL 不回滚，会两库不一致、残留脏数据。这是我知道的一致性缺口。要补的话可以加处理状态标记 + 失败清理，或者用"标记-确认"两阶段。

**追问：那你怎么保证最终一致？** → 现在没完全保证，是已知边界。可行方案：文档有 status 字段，只有两库都写成功才置 READY，失败的定期扫描重试或清理。

## 七、安全：密码存储 / 注入防护

**八股**：密码不能明文/不能可逆加密，要用加盐哈希（BCrypt）；用户输入拼进 SQL/shell/路径都要防注入。

**我项目里**：
- 密码用 **BCryptPasswordEncoder** 哈希存储（BCrypt 自带盐、可调 cost）。
- 文件名 sanitizeFileName 拒绝 `..`/绝对路径，防目录穿越。
- 所有拼进 shell 命令的路径用**单引号转义**（shellQuote），Office 预览路径用 `/home/gem/` 白名单前缀。
- MCP 用户配置做 SSRF 校验（禁私网、解析 IP 拦截）。

**追问：BCrypt 和 MD5+盐 区别？** → BCrypt 慢（可调 cost 抵抗暴力破解）、每次自动生成盐并存进 hash；MD5 太快、适合被 GPU 爆破，不适合存密码。
