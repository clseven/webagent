# 边界与异常处理问答

> 用途：面试官很少真读源码，但一定会问"边界有没有处理"——token 超了怎么办、用户卡了怎么办、并发怎么办、挂了怎么办。
> 这份文档把项目里真实存在的边界和坑，翻译成可以直接背的业务化应答。
> 
> **核心策略**：每个点都按「现状怎么做 → 坑在哪 → 正确该怎么做」三段说。面试官最烦"不知道有边界"，最认可"知道边界并能说清代价"。主动认边界 > 被戳穿。

## 0. 万能话术（背下来）

被问到没做的点时，用这句兜底，把"没做"包装成"有意识的取舍"：

> 这个点我当时优先级排在后面，属于我明确知道的边界，但为了先跑通主链路做了取舍。如果要上生产，这是我会第一批补的。

结尾自我定位（主动交代边界，掌握节奏）：

> 这个项目我把它定位成一个能完整跑通的单机 demo，主链路是通的。真正的短板集中在三类：并发（锁、超时中断、线程池）、状态外置（多实例、重启恢复）、数据一致性（异步事务）。这三类我都清楚现状和该怎么补。

## 1. 上下文 / Token 边界（最高频）

### Q：对话越来越长，token 超了怎么办？

> 我做了上下文压缩。每轮估算消息总 token，超过阈值就把最旧的一批消息交给 LLM 生成摘要，用摘要替换原文，保留最近约 40% 的原始对话作为上下文。
> 有个细节：切分点会对齐到工具调用边界，避免把一组 tool_call 和它对应的 tool_result 切散——切散了模型会直接报协议错误。这个坑我踩过所以专门处理了。

### 追问：token 怎么估算的？准吗？

> token 我分两个地方处理，用途完全不同，不是一回事：
> 一是**真实用量**——每次 LLM 响应里都有 usage 字段（prompt_tokens、completion_tokens、total_tokens），我都取了，存进 TokenUsageEntity 做成本统计、按模型和按天的用量分析，这个是精确的。
> 二是**压缩的触发判断**——这个必须在请求**发出前**做，而真实 usage 要等响应回来才有、时机对不上，所以这里用字符数除以经验比例（中文约 3 字符/token）快速预估，超过约 2.4 万就触发压缩。
> 这里我是**故意不做精确的**：压缩触发只是个开关，不是账本，阈值也留了保守余量，估个大概完全够用，估偏一点顶多是压缩早触发或晚触发一轮，不影响正确性。为了一个开关去引 tokenizer、每轮多算一遍，不划算。该精确的地方（记账）精确，该省的地方（开关）省，是有意的取舍。

> **为什么压缩处不能直接用响应里的真实 usage？** 因为时机对不上：真实 usage 是"这次调用花了多少"，响应回来才知道；压缩要判断的是"这批消息发出去之前要不要先压"，是发送前的预算控制。你不可能用上一次的真实 usage 去精确预测这一次加了新消息和工具结果之后的大小。所以两者各管一段——真实 usage 管事后记账，字符估算管事前触发。

### 追问：压缩本身失败了怎么办？

> 目前失败会降级成保留原文。坦白说这里兜底不够——压缩失败等于既没省下上下文、又多花了一次 LLM 调用，上下文照样可能爆。更好的做法是压缩失败时按最近 N 条消息硬截断保底，保证请求一定能发出去。

### 追问：摘要那次 LLM 调用的 token 算进统计了吗？

> 主链路的 token 都统计了，但摘要这次调用、还有意图判断那次调用的 token 目前没单独计入。这是统计口径的一致性缺口，要补的话应该把所有 LLM 调用都纳入 token 账本。

### Q：单个工具返回结果特别大（比如读了个几万行的文件）怎么办？

> 分两种情况。read_file 我改成了按行窗口分页，单次 observation 是有界的，不会一次性把整个大文件塞进上下文。但 shell 命令、MCP 工具、文档解析这类一次性大输出，目前只有一个告警 Hook 提示、没有真正截断。head+tail 截断（保留头尾、砍中间）是我列在 TODO 里的已知缺口，因为大结果确实可能一次性撑爆上下文。

## 2. 用户等待 / 卡住 / 超时边界

### Q：Agent 执行很久，用户一直等着怎么办？

> 三层处理：
> 1. 前端走 SSE 流式，plan、thinking、工具调用、observation 实时往前端推，用户能看到执行进度，不是对着空白干等。
> 2. ReAct 循环有最大迭代次数上限，到顶会返回提示，不会无限循环烧钱。
> 3. 用户可以主动停止——后端通过检测客户端断开（emitter 的 cancel 状态）在循环开始时中断，实现即时停止。

### 追问：到了最大迭代次数、任务还没做完呢？

> 现在是直接返回一句"我尝试了多次仍未完成，请简化要求或提供更多信息"。这个体验其实不够好，因为中间产物就丢了。理想做法是保留已完成的步骤、允许用户接着上一次继续，而不是从头再来。这是可优化点。

### 追问：子代理（run_subagent）跑超时了怎么办？

> 子代理有超时时间，超时后主流程不再等它、直接往下走。
> 这里有个我清楚的坑：超时用的是 CompletableFuture 的 orTimeout，它只让主线程不等了，但**底层那个正在跑 LLM 循环的线程并没有被真正中断**，它还会在后台继续调模型、继续烧 token。正确做法是传一个取消信号进去、用支持中断的执行器，让超时能真正停掉子任务。另外它跑在默认的 ForkJoinPool 上，而子代理是阻塞等 LLM 的，严格讲应该用独立的线程池，避免占满公共池影响别的任务。

### Q：调用大模型的接口挂了 / 超时 / 限流怎么办？

> 有一套错误处理：错误码映射、区分可重试和不可重试的错误、流式和非流式都有重试边界。
> 另外几个外部依赖都做了降级，不让单点故障拖垮整条链路：
> - Rerank 服务（本地 BGE）不可用 → 退回用向量相似度分数排序。
> - Query Rewrite 不可用 → 用原始 query 检索。
> - Milvus 关闭或不可用 → 检索退化为空结果，但不影响对话继续。

### 追问：SSE 连接中途断了怎么办？

> 目前断开后不能自动重连，需要用户重新发消息。要产品化的话应该加断线重连 + 事件续传（记录 last-event-id），这是前端侧的已知缺口。

## 3. 并发边界（Java 后端必被深挖）

### Q：同一个用户同时发两个请求、或者同时开两个会话，会重复创建沙箱吗？

> 设计上一个用户复用一个沙箱。创建时做了三层防护：按用户加锁 + 进入临界区后双重检查 + 数据库记录兜底。
> 说实话我这个锁的实现有瑕疵：锁对象是从一个 map 里按 userId 取的，但用完在 finally 里又把它从 map 删了。这会有一个窗口——一个线程删掉锁对象后，另一个线程重新 computeIfAbsent 会拿到一个新的锁对象，两个线程锁的不是同一个对象，锁就失效了。目前靠双重检查和 DB 记录兜住了，实际没出过问题，但严格讲应该改成不回收锁对象、或者用 userId.intern() 做锁、或者用分段锁。这是我知道要改的。

### Q：这个系统能部署多台机器做水平扩展吗？

> 现在不能，这是它最大的边界，我很清楚。
> 原因是关键状态全是内存态：用户↔沙箱的映射、对话摘要、TodoState、沙箱视图 token，全在内存的 Map 或实例字段里；锁也只在单个 JVM 内有效。
> 多实例部署会出问题：请求被负载均衡打到另一台，那台内存里没有这个用户的沙箱映射，会重复建沙箱；synchronized 锁跨不了实例，并发创建也拦不住。
> 要上生产，得把这些状态外置到 Redis / 数据库，锁换成分布式锁（比如 Redisson）。这是明确的 roadmap。

### Q：服务重启会发生什么？

> 沙箱的绑定关系（用户、sandboxId、endpoint）存在数据库，重启时会从库里恢复，并用健康探针检查 endpoint 是否还活着，不健康就清理记录重建。
> 但内存态的东西会丢：对话摘要、TodoState、沙箱视图 token。用户表现是需要重新请求视图 URL、多轮对话的压缩上下文重来。核心业务数据不丢，运行时状态会丢。

### Q：异步创建沙箱，如果创建失败前端怎么知道？

> 创建会话时沙箱是异步建的，为的是不阻塞前端拿到会话响应。这里两个点：
> 一是异步任务里我特意用 sessionId 查库拿 userId，没依赖 ThreadLocal——因为 ThreadLocal 跨不了线程，这个坑我避开了。
> 二是坦白讲，异步创建失败目前只记日志，没有把失败状态回推给前端。用户会看到会话建好了、但实际发消息时沙箱没就绪。要补的是一个沙箱状态字段 + 前端轮询或推送。

## 4. 多用户隔离 / 安全边界

### Q：怎么保证用户 A 看不到用户 B 的数据？

> 所有业务数据都带 userId。会话、上传文件、知识库、沙箱、MCP 配置，访问时都校验归属当前用户。
> 诚实说：系统性的越权测试矩阵我还没做全。像 docId、sessionId、appId 这类直接从参数传进来的接口，需要专门的越权用例覆盖（拿别人的 id 试探）。我把它列在了要补的渗透测试清单里。

### Q：用户上传文件，怎么防路径穿越攻击（比如文件名带 ../）？

> 文件名做了清理：把反斜杠统一成正斜杠、只取最后一段路径名、拒绝空名和 . 与 ..，所以 ../ 这种穿越是挡住的。
> 边界上还能加强的：控制字符和空字符过滤、Windows 保留名（CON、NUL 这些）处理。另外路径里除了文件名还有 kbId 这类 id 段，那些 id 是后端生成的、不是用户直接可控的，所以没穿越风险——但如果哪天 id 变成用户可传，也要一起校验。

### Q：动态 MCP 让模型能连接外部地址，不怕 SSRF 吗（比如让它去访问内网）？

> 做了防线：用户配置的 MCP 默认禁 HTTP 只允许 HTTPS、禁私网。校验时会解析目标域名的 IP，把环回、私网、链路本地、site-local、组播、云元数据网段（169.254）这些都拦掉；shell 类型的 MCP 只能在用户自己的沙箱里跑、不在宿主机执行，还限制了命令长度和环境变量格式。
> 更深一层有个我知道但还没堵的风险——DNS rebinding：我在"配置校验时"解析一次 DNS 判断是不是内网，但"真正发请求时"会再解析一次，攻击者可以让域名第一次解析成公网、第二次解析成内网地址来绕过。严格防要在请求层用校验时已经解析定的那个 IP，或者禁止跟随重定向。这个是 TOCTOU 类问题，我意识到了。

### Q：沙箱视图的 token 泄露了怎么办？

> token 是短期有效的、只存在内存、服务重启就失效。而且 token 里不直接存 endpoint，只存能定位用户的信息，每次代理请求都按 userId 查最新 endpoint——这样沙箱重建 endpoint 变了也能恢复。
> 残留风险是：有效期内如果 token 泄露，别人能访问该用户的沙箱视图。可以加绑定 user-agent、绑定 sessionId、或者一次性 nonce 来增强。

## 5. RAG / 数据一致性边界

### Q：文档上传解析失败，会留下脏数据吗？

> 会，这是我明确知道的一致性问题。文档处理是异步的，而且没有事务包裹，所以如果在写完切片、写完向量之后的某一步失败了，前面写进去的切片和向量就残留了。要补的是失败回滚、或者用一个处理状态字段标记失败后清理。

### 追问：还有别的一致性缺口吗？

> 有两个：
> 一是 READY 状态不等于沙箱已同步——文档标成 READY，但同步到沙箱那步失败只会把 sandboxSynced 置 false，状态还是 READY，会误导。
> 二是用户注销时删业务数据，但不清理沙箱里的文件，会留孤儿文件。
> 这些都是我盘过的、列在已知风险里的点。

### Q：检索不到内容 / 向量库是空的怎么办？

> 本地开发默认把 Milvus 关了，方便快速启动，这种情况下检索会退化成空结果、但对话本身不受影响。要真正演示 RAG 效果得开 Milvus 并确认集合里有向量数据。降级时怎么给用户一个"资料库未就绪"的提示，是要补的体验点。

### Q：RAG 能解决模型幻觉吗？

> 不能"解决"，只能"降低"无依据回答的概率。手段是：Query Rewrite 提升召回、Rerank 降低噪声、把 topK 切片作为上下文注入、再用 Prompt 约束模型优先依据资料回答、资料不足时明确说资料不足。幻觉不可能靠 RAG 完全消除，还得靠 Prompt 约束和检索质量。

## 6. 代码质量 / 可维护性（偶尔会问）

### Q：你说做了服务拆分，具体拆了什么？

> 原来 AgentServiceImpl 一个类要管上下文准备、规划、工具装配、MCP、RAG、Hook、模型调用、消息保存、SSE 转发，太重。我拆成了：AgentTurnContextService 准备上下文、AgentPlannerService 做规划、ReactAgentFactory 造执行器、ReactAgentHookService 注册 Hook，入口只留编排、校验、消息保存。这样同步调用和 SSE 流式能复用同一套上下文和执行器构造。

### 追问（如果他看过代码）：那流式那个主方法怎么还那么大？

> 对，chatStream 那条流式主路径确实还偏重，认知复杂度偏高。拆分主要拆的是"横切职责"，但事件流转本身还集中在一个方法里。下一步应该按事件类型拆成独立的 handler。这是我承认的技术债。

## 7. 速查表：一句话现状

| 边界问题 | 现状 | 最大的坑 | 该怎么补 |
| --- | --- | --- | --- |
| token 超长 | LLM 摘要压缩，对齐工具边界 | 字符数粗估不准；压缩失败无保底 | 接 tokenizer；失败硬截断 |
| 大工具输出 | read_file 分页 | shell/MCP 输出只告警不截断 | head+tail 截断 |
| 执行太久 | SSE 进度 + 迭代上限 + 可停止 | 到上限任务丢失 | 保留中间产物、可续跑 |
| 子代理超时 | orTimeout 主流程不等 | 后台线程不真中断、烧 token | 传取消信号 + 独立线程池 |
| LLM 故障 | 错误映射 + 重试 + 降级 | — | 已较完整 |
| 重复建沙箱 | 锁 + 双检 + DB 兜底 | 锁对象被回收、锁可能失效 | 不回收锁 / intern / 分段锁 |
| 水平扩展 | 不支持 | 状态全内存、锁单 JVM | 状态外置 Redis、分布式锁 |
| 服务重启 | 沙箱关系从 DB 恢复 | 摘要/todo/token 丢失 | 运行态持久化 |
| 异步建沙箱失败 | 记日志 | 失败不回推前端 | 状态字段 + 推送 |
| 多用户越权 | 都带 userId 校验 | 越权测试矩阵不全 | 系统性渗透测试 |
| 路径穿越 | 文件名清理挡 ../ | 控制字符/保留名未处理 | 补字符过滤 |
| MCP SSRF | 禁 HTTP/私网 + 解析 IP 拦截 | DNS rebinding（TOCTOU） | 请求层用已定 IP |
| 视图 token 泄露 | 短期、内存、按 userId 查 | 有效期内泄露可访问 | 绑定 UA/nonce |
| 文档处理失败 | 异步处理 | 无事务、脏数据残留 | 回滚或状态标记清理 |
| RAG 空结果 | Milvus 默认关 | 检索退化为空 | 开 Milvus + 灌数据 |
