# 技术选型与场景设计问答

> 用途：面试中场到后半段的两类主战场。
> 第一部分「为什么不用 XXX」——考选型判断，面试官想听的不是"哪个好"，而是"你为什么在你的场景选了这个、代价是什么"。
> 第二部分「如果要…你怎么做」——考架构思维，是开放题，没有标准答案，考的是思路是否清晰、有没有分层。
>
> **通用打法**：选型题一律用「场景需求 → 候选方案 → 我的选择 → 代价/取舍」四段；场景题一律先说"先明确目标和约束"，再分层给方案，别一上来就堆技术名词。

# 第一部分：技术选型「为什么不用 XXX」

## Q：为什么自己撸 ReAct 循环，不用 LangChain / LangGraph？

> 主要是**可控性**。ReAct 的循环逻辑本身不复杂——调模型、拿到 tool_call 就执行、把 observation 塞回去接着调，直到出最终答案。真正的复杂度在我自己的业务需求上：多轮 tool_call 的调度、Hook 注入（图片观察、最终回答门禁）、上下文压缩、SSE 双层流式、token 统计。这些用框架反而要去适配它的抽象，出问题也难调。
> 自己实现的代价是：通用能力得自己补，比如更复杂的 agent 编排模式。但对一个需求明确的项目，我宁愿要一个我完全掌控、能逐行 debug 的循环，而不是被框架的抽象绑住。这是可控性和开发速度之间的取舍。

## Q：为什么用 Spring AI 而不是直接调 HTTP / 用 LangChain4j？

> （简历里票联行写的是 Spring AI）选 Spring AI 是因为它贴 Java 后端生态、和 Spring 的依赖注入、配置体系融得好，Tool Calling 的封装也够用。
> WebAgent 这边我是自己封装的 LLM 调用层（BaseLlmServiceImpl），因为我要统一多个模型（DeepSeek 做规划执行、Agnes 做视觉）的请求响应格式、错误码映射、重试、流式解析、token 统计——这些我要完全自己控制，所以做了一层自己的抽象，不同模型实现同一个接口。

## Q：为什么 SSE 不用 WebSocket？

> 场景是**服务端单向往前端推**执行过程（plan、thinking、工具调用、observation、answer），不需要前端持续往回发。SSE 就够了，比 WebSocket 轻，基于普通 HTTP、天然支持断线重连语义、后端用 Flux 就能实现。
> 但我没用浏览器原生的 EventSource，因为它只支持 GET、不方便带请求体和复杂鉴权。我前端用 fetch + ReadableStream 手动解析 SSE，这样能发 POST、能带完整的对话参数和 token。
> 代价：SSE 是单向的，如果以后要做前端实时打断、双向协作，就得换 WebSocket。但目前的交互模型 SSE 是最合适的。
> （注：沙箱视图那块——noVNC、terminal——用的是 WebSocket，因为那是真正的双向交互，我后端做了 WebSocket 代理桥接。所以其实两个都用了，用在各自合适的地方。）

## Q：为什么沙箱做成用户级，不是会话级？为什么不用 K8s Pod？

> 从会话级改用户级，是因为会话级有三个痛点：多个会话之间文件和环境不能复用、同一用户跨会话工作区割裂、endpoint 变了恢复麻烦。改成一个用户复用一个长期沙箱后，多会话共享文件、浏览器、工作区状态，通过 userId 就能查到最新的 sandboxId 和 endpoint 来恢复。
> 没上 K8s 是因为当前是单机部署，用的是 AIO Sandbox（一个集成了 shell、文件、浏览器、noVNC、code-server 的容器镜像），够用且轻。如果要规模化、要真正的资源隔离和弹性调度，K8s Pod（每用户一个 Pod、配 resource limit）是自然的演进方向，但那是水平扩展阶段的事。

## Q：为什么规划和执行要拆成 PlanAgent 和 ReactAgent 两个？

> 职责不同。PlanAgent 是**任务前规划**——根据用户输入、历史、可用工具，先产出目标、执行策略、成功信号和初始步骤，它不碰工具。ReactAgent 是**执行器**——维护 tool_call → observation → 推理的循环。
> 拆开的好处：一是避免 ReactAgent 一上来就盲目调工具，先有个大局；二是前端能先展示 plan 事件，用户先看到"它打算怎么做"；三是两个阶段可以用不同的 prompt 甚至不同的模型。
> 代价是多一次 LLM 调用（规划那次）。但对复杂任务，先规划带来的执行质量提升值这个成本；简单任务我有轻量路由（LightweightChatRouter）可以跳过重规划。

## Q：为什么 RAG 用向量检索，不直接把知识微调进模型？

> 微调的问题：成本高、知识一更新就得重训、而且微调是把知识"焊死"进权重，无法溯源、无法按用户隔离。
> RAG 的优势正好对应我的场景：知识库是多用户的、随时上传更新的、要能说清答案来自哪篇文档的。向量检索能做到即时更新、按 userId 隔离、可溯源。
> 代价是检索质量直接决定效果，所以我加了 Query Rewrite 提召回、Rerank 降噪。微调和 RAG 其实不互斥——微调改"模型的能力和风格"，RAG 补"实时的、私有的知识"，我的场景要的是后者。

## Q：为什么规划执行用 DeepSeek，视觉用 Agnes，分开两个模型？

> 术业有专攻 + 成本。DeepSeek 在中文推理和工具调用上性价比高，适合做规划和执行这种纯文本推理的主力。视觉观察（看浏览器截图、理解页面）是另一种能力，用专门的视觉模型 Agnes。
> 我的 LLM 调用层做了抽象，不同模型实现同一接口，所以"哪个任务用哪个模型"是可配置的。这也为以后做模型路由留了口子——比如简单任务用便宜模型、复杂任务用强模型。

## Q：为什么向量库用 Milvus？（且为什么默认关闭）

> Milvus 是成熟的开源向量库，支持大规模向量、多种索引，生产可用。
> 默认关闭纯粹是**本地开发便利**——关掉它整个应用能快速起来，不用每次都拉起 Milvus。这是开发配置不是生产状态，真正演示 RAG 得开 Milvus 并确认集合里有数据。这一点我在文档里明确标了，避免被误解成"RAG 没做完"。

## Q：你提到 prompt caching，那不是模型厂商自带的吗？你做了什么？

> 对，缓存本身是厂商实现的（DeepSeek、Anthropic 都有），我不需要自己做缓存。**但缓存是厂商的、命中率是我的责任**——能不能命中，取决于我怎么组织请求。
> 缓存是按 token 前缀**逐字节匹配**的：只要前面有一个字符变了，从那个字符往后全部 miss、重新计费。所以要命中，我要保证请求的前缀部分**稳定不变**：
> 1. 把 system prompt、工具定义、历史消息这些**不变的内容放前面**，作为固定前缀；每轮唯一变化的新用户输入放最后。这样重复的前缀能命中缓存，既省 token 又降首字延迟。
> 2. 顺序不能乱——工具列表、JSON 字段顺序每轮要一致，否则序列化出来的字符串变了，内容一样也 miss。
> 我代码里两个设计就是在维护这个前缀：一是 run() 里把 history 作为**固定前缀**传入（注释就写了"利用 prompt caching"）；二是上下文压缩生成的摘要**特意不塞进 system prompt**、而是作为普通消息放回历史，避免破坏 system 这段稳定前缀。
> 一句话：厂商给了缓存这个能力，但要真正吃到降价，得靠我在请求组织上保证前缀稳定。

## Q：MCP 为什么要接？和普通工具有什么区别？

> 普通工具是写死在后端代码里的，加一个工具要改代码、发版，扩展成本高。MCP 是一个标准协议，让外部工具能用统一方式暴露给 Agent，系统运行时动态发现、接入，不用改代码。
> 关键是：MCP 工具进系统后不是特殊通道——我把 MCP Server 返回的工具包装成 RealMcpTool，进和本地工具**完全一样**的调用流程。ReactAgent 根本不关心工具来自本地还是 MCP，按统一定义调用就行。这样扩展性上来了，执行路径还是一套。

# 第二部分：场景设计「如果要…你怎么做」

## Q：如果要支持 10 万用户 / 水平扩展，你怎么改？

> 先定位瓶颈：当前最大的阻碍是**状态全是内存态**，所以不能多实例。我会分几步：
> 1. **状态外置**：用户↔沙箱映射、对话摘要、TodoState、沙箱视图 token，从内存 Map 迁到 Redis / 数据库，让任意一台实例都能读到。
> 2. **锁换成分布式锁**：现在的 synchronized 只在单 JVM 有效，创建沙箱这种要跨实例互斥的，换成 Redisson 分布式锁。
> 3. **沙箱资源池化**：一个用户一个长期容器，10 万用户就是 10 万容器，扛不住。要么做沙箱的按需创建 + 空闲回收（LRU 淘汰长期不活跃的），要么上 K8s 用 Pod 调度 + 资源配额。
> 4. **无状态化应用层**：应用层做成无状态，前面挂负载均衡；SSE 这种长连接要注意会话亲和（sticky session）或用支持长连接的网关。
> 5. **异步和削峰**：建沙箱、文档处理这些重操作走消息队列，避免瞬时打满。
> 优先级上，我会先做状态外置 + 分布式锁（这是能不能多实例的门槛），再做沙箱池化（这是成本和资源的门槛）。

## Q：如果要新加一个工具，完整链路是什么？

> 分两种：
> **本地工具**（要写代码）：实现 Tool 接口，定义好 name、描述、参数 schema（这个 schema 会给模型看，让它知道怎么调），注册进工具集合。模型在 ReAct 循环里就能选它、调它，返回 observation。核心是参数 schema 要写清楚，因为模型是靠它决定怎么调的。
> **外部工具（走 MCP，不用写代码）**：我实现的是一套**通用的 MCP 客户端能力**——不针对任何具体 server。要接入某个外部工具，只需要在配置里加一条 server（一个 URL 或一段 shell 命令），系统运行时动态连接、拉取它暴露的工具列表、包装成 RealMcpTool，进和本地工具完全一样的调用流程。
> 换句话说：MCP 客户端是我一次性做好的通用能力，"加一条 server 配置"是运行时的使用动作。正因为客户端是通用的，接任意 MCP server 才不用改代码。

（补充见下方"MCP 到底是通用的还是要逐个配"）

> 要注意的边界：工具参数不能完全信任模型生成的，要校验；有副作用的工具（写文件、执行命令）要考虑并发和幂等；输出可能很大要考虑截断。

## Q：你说 MCP 是通用的，那"配一个 server"又是什么意思？不矛盾吗？

> 不矛盾，是"能力"和"使用"两个层次：
> - **通用的是 MCP 客户端能力**（我做的）：连接管理、工具发现、包装成统一 Tool、命名冲突处理、SSRF 安全校验——这套对任何符合 MCP 协议的 server 都通用，做一次就行。
> - **"配一个 server"是运行时使用**（用户/管理员做的）：想接哪个外部工具，就往配置里加一条它的地址或启动命令。
>
> 打个比方：我写的是浏览器（通用），用户输入的网址是配置。浏览器通用，但看某个网站得输网址；MCP 客户端通用，但接某个具体工具得配一条 server。正因为通用，才能靠"填配置"接入任意 server 而不改代码。
> 具体实现上：用户配置存在沙箱里的 `/home/gem/.mcp/servers.json`，按用户隔离；我还做了几个工具（add/list/remove server）让 Agent 能自己管理这些配置；真实连接和调用由官方 MCP SDK 驱动的 McpClientManager 完成。

## Q：如果要做多模型自动路由（简单问题用便宜模型，复杂任务用强模型）怎么做？

> 我已经有基础——LLM 调用层做了抽象，多模型实现同一接口，模型是可配置的，也有个轻量路由（LightweightChatRouter）判断是不是简单对话。
> 要做完整路由，我会加一个**路由决策层**，在正式执行前先判断任务复杂度。判断依据可以是：意图分类（闲聊 vs 任务）、是否需要工具、历史长度、是否涉及多步。
> 决策方式有两种：规则法（快、免费，但覆盖不全）或用一个小模型做分类（准，但多一次调用）。我会先用规则兜大部分，边界情况再上小模型。
> 关键取舍：路由判断本身有成本（如果用模型判断），要保证"省下的钱 > 判断的成本"，否则得不偿失。而且要有兜底——路由错了（简单问题路给了弱模型答不好），要能升级重试。

## Q：如果 token 成本要砍一半，你从哪下手？

> 按性价比排：
> 1. **prompt caching**（几乎零成本收益最大）：保证 system prompt、工具定义、历史消息作为稳定前缀，让厂商缓存命中，重复前缀部分大幅降价。我现在已经把 history 作为固定前缀传了，就是为这个。
> 2. **上下文压缩调优**：更激进的压缩阈值、更狠的摘要，减少每轮带的历史量。
> 3. **模型路由**：简单任务下沉到便宜模型（见上一题）。
> 4. **工具输出截断**：大的 observation 做 head+tail 截断，别把几万行日志整个塞进上下文（这也是我的 TODO）。
> 5. **减少不必要的 LLM 调用**：比如能用规则判断的意图，就别用一次模型调用。
> 我会先上 prompt caching 和输出截断，这两个是"不损失效果的纯优化"；模型路由和激进压缩要权衡效果损失，放后面。

## Q：如果要保证 Agent 执行的可观测性（线上出问题能排查）怎么做？

> 三个层面：
> 1. **结构化日志 + traceId**：一次任务从进来到结束串一个 traceId，每一步（规划、每轮工具调用、每次 LLM 调用）都带上，能还原完整执行链。（注：我实习时踩过并发日志顺序错乱的坑，就是异步链路里日志时序对不上，所以我知道这里要用事件时间戳而不是写入时间来排序。）
> 2. **关键指标**：每轮迭代数、每次 LLM 调用的 token 和耗时、工具调用的成功率和耗时、压缩触发次数。token 我已经存了 TokenUsageEntity。
> 3. **执行链回放**：我把每次任务的事件列表（plan/tool_call/observation/answer）都存了，出问题能回放当时模型看到了什么、怎么决策的——这对排查"模型为什么这么答"特别关键。

## Q：如果模型一直调用工具、陷入死循环怎么办？

> 已经有的：最大迭代次数上限，到顶强制停止返回提示，防止无限循环烧钱。
> 可以加强的：检测**重复调用**（同样的工具同样的参数连续调）直接打断；检测"无进展"（连续几轮没产生新信息）提前收尾；给单次任务设总 token 预算，超了就停。
> 更根本的是 prompt 层面——让模型清楚什么算完成（PlanAgent 产出的成功信号），配合最终回答门禁（TodoState）判断是不是真做完了，减少无意义的循环。
