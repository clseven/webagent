# 变更 2026-07-14 RAG 重排与 Agent 续接同步

## 背景

本次同步发生在 `c3f249d` 到 `19cbc86` 之间多轮提交之后。代码层面统一了知识库检索链路、引入了 DeepSeek 独立重排、新增了 Agent 续接检查点和单轮时间快照，但 `knowledge-base/` 笔记仍停留在 2026-07-06 / 07-09 版本，多处描述与实际实现不一致。

## 本次覆盖的代码事实

### RAG 检索与重排

- `RerankService` 抽象新增 provider 切换，生产默认使用 DeepSeek `deepseek-v4-flash`，BGE 保留为可切换实现。
- 新增 `DeepSeekRerankServiceImpl`，使用独立 `RAG_RERANK_MODEL` 调用 OpenAI 兼容 chat completion 接口，不复用主流程 executor。
- 新增 `RerankResult`，显式标记 `reranked=true/false`，区分模型重排分数和向量降级分数。
- 新增 `RerankResultFilter`，成功重排时按 `min-score` 过滤后截取 topK，向量降级时只截取 topK。
- 旧的 `retrieve.min-score` 不再作为最终相关度门槛，最终门槛改由 `rerank.min-score`（默认 0.8）承担。
- REST 单库搜索、`knowledge_search` 工具和 Agent 自动预检索统一委托 `KnowledgeService` 的多知识库搜索入口。

### Agent 续接

- ReAct 最大迭代数 200，达到上限返回 `AgentRunStatus.PAUSED_MAX_ITERATIONS`。
- 新增 `AgentRunCheckpoint`，保存协议级模型消息顺序和工具调用关联，不保存前端事件、文件附件或图片 base64。
- `ConversationServiceImpl` 保存助手消息时写入 `runStatus` 和 `checkpointJson`。
- 下一轮 `AgentTurnContextService.prepare` 优先从检查点恢复历史；旧数据降级为从展示事件生成续接说明。

### 时间感知

- 新增 `AgentTimeContext` 单轮不可变时间快照，注入规划器、执行器、社交轮和子代理。
- 新增 `AgentTimeContextService`，在 `prepare` 开始时创建快照。
- 新增 `CurrentTimeTool`（工具名 `current_time`），用于获取调用时刻新时间或指定 IANA 时区。
- 配置项 `agent.time-zone`（默认 `Asia/Shanghai`）控制快照时区。

### 请求级知识库开关

- `ChatRequest` 新增 `knowledgeEnabled`（默认 true），流式入口从 query 参数读取。
- `knowledgeEnabled=true` 且应用关联知识库时执行自动预检索并暴露 `knowledge_search` 工具。
- `knowledgeEnabled=false` 时不注入知识库上下文，也不配置知识库工具。
- 知识库自动检索不再由 `TurnPolicy` / SOCIAL 轮次策略控制。

### 模型配置

- 主流程 planner/executor 模型从 `deepseek-v4-flash` 切换到 `deepseek-v4-pro`。
- Rerank 模型独立配置为 `deepseek-v4-flash`，不继承主流程模型。

## 后续建议

- 如果后续新增检索入口，应复用 `KnowledgeService` 统一搜索，避免三条链路行为漂移。
- 如果后续需要调整重排 provider，优先通过环境变量 `RAG_RERANK_PROVIDER` 切换，不要在代码里硬编码。
- 如果后续新增 TurnPolicy 开关，不要把知识库启停塞回去，知识库已由请求级 `knowledgeEnabled` 控制。
