# Obsidian 项目知识库设计

## 目标

为 `webagent-clean` 代码仓库建立并持续维护一套 Obsidian 项目知识库。

知识库的主要读者是刚开启新会话的 AI 助手。它需要帮助 AI 快速建立准确的项目认知，根据当前任务找到相关模块，并在需要强证据时回到源代码验证。同时，这套结构也应方便开发者浏览和理解。

第一阶段先对现有仓库进行一次完整整理。该流程在真实项目中验证并调整后，再提炼为可复用的初始化 Skill 和持续维护 Skill。

## 语言规范

- 知识库正文、标题、说明、风险、问题和变更记录默认使用简体中文。
- Java 类名、方法名、配置键、HTTP 路径、数据库字段及其他代码标识符保留原文，并使用反引号标记。
- 文件名以清晰的中文名称为主，例如 `RAG 模块.md`、`用户请求处理流程.md`。
- 常用英文名、缩写和代码名称通过 `aliases` 属性记录，便于搜索和创建链接。
- 首次出现的重要术语可采用“中文名称（English Term）”形式，后续使用中文简称。
- 不翻译会造成歧义的技术名称，例如 Spring AI、SSE、RAG、MCP、OpenAPI。
- 引用项目现有中文或英文文档时，应忠实表达原意，不为了统一语言而改变技术含义。

## 建设目标

- 在架构层面覆盖整个仓库。
- 深入整理核心模块和重要跨模块流程。
- 充分利用 Obsidian 的文件夹、双链、反向链接、局部关系图、属性、Bases 和搜索能力。
- 让重要结论可以追溯到代码或项目权威文档。
- 明确区分已验证事实、分析推断和可能过期的知识。
- 根据 Git 变更持续增量维护，避免每次重新扫描整个仓库。
- 不维护第二套容易过期的人工架构图。

## 非目标

- 不为每个类或方法单独生成笔记。
- 不把代码目录原样复制到 Obsidian。
- 不把 Obsidian 关系图称为自动生成的 Java 调用图。
- 不要求使用 Excalidraw 维护架构图。
- 不让核心导航依赖 Dataview 或其他社区插件。
- 不用知识库取代源代码、测试、API 契约和仓库内的权威文档。
- 不在真实初始化流程得到验证前创建通用 Skill。

## 存储位置

目标 Obsidian Vault：

```text
D:\obsidian
```

项目知识库目录：

```text
D:\obsidian\Projects\WebAgent Clean
```

实际写入前需要确认 `D:\obsidian` 是有效的 Obsidian Vault。由于该目录位于当前代码仓库工作区之外，写入时需要获得明确的文件系统授权。

Obsidian Vault 是知识库的唯一主要存储位置。代码仓库中只保留本设计文档，不重复保存一份生成后的知识库。

## Obsidian 原生能力的使用方式

### 文件夹

文件夹提供稳定的物理分类和可预测路径，回答“某类知识存在哪里”。文件夹不负责表达所有系统关系。

### 内部链接

内部链接负责表达已确认的项目关系：

- 模块页链接其上游模块和下游模块。
- 流程页链接所有参与模块和使用的契约。
- 决策页链接受影响的模块和流程。
- 关键组件页链接所属模块和直接概念依赖。
- 风险页和开放问题页链接它们所影响或质疑的知识。

链接必须出现在有明确语义的正文或关系字段中。不能只为了让关系图显得连接丰富，就让所有页面机械地链接项目入口。

### 反向链接与未链接提及

反向链接用于查看哪些知识依赖当前页面。未链接提及用于发现页面标题或别名已经出现在其他笔记中，但尚未建立链接的情况。

页面别名可包含代码名称、缩写及中英文替代名称。例如中文模块页可以把主要 Java 服务类名作为别名。

### 关系图

局部关系图是主要的关系可视化方式。阅读模块或流程时，通常查看深度为一层或两层的局部关系图。

全局关系图作为辅助视图，并应筛选到当前项目。可以通过路径、属性或标签为模块、流程、契约、决策、风险和未验证内容设置不同颜色组。

变更记录和待整理页面主要承载历史与临时信息，应尽量从默认项目关系图中排除，避免干扰稳定架构。

关系图展示的是知识页面中已经记录的关系，不是实时代码调用图。

### 属性

所有生成的知识页面采用少量统一属性：

```yaml
---
project: webagent-clean
type: module
area: rag
status: verified
updated: 2026-06-13
source_commit: <Git 提交>
aliases:
  - <可选别名>
---
```

必需属性：

- `project`：稳定的项目标识。
- `type`：页面类型，例如 `architecture`、`module`、`component`、`flow`、`contract`、`decision`、`risk`、`question`、`change` 或 `staging`。
- `area`：页面所属架构领域，无明确归属时可省略。
- `status`：`verified`、`inferred` 或 `stale`。
- `updated`：页面最近一次复核日期。
- `source_commit`：最近一次复核所依据的 Git 提交。

属性键保留稳定的英文值，便于 Obsidian 查询、Skill 处理和跨工具兼容。面向用户展示的列名和说明使用中文。

只有当新属性确实支持某个视图或维护判断时才添加，避免元数据膨胀。

### Bases 与 Dataview

优先使用 Obsidian 原生 Bases 创建动态视图，因为其数据直接来自 Markdown 属性，不依赖社区插件。

第一版至少提供：

- 全部稳定项目知识。
- 按领域分组的模块列表。
- 跨模块流程列表。
- 需要复核的推断或过期页面。
- 最近复核的页面。

Dataview 仅作为补充，用于 Bases 不便表达的查询。即使关闭 Dataview，核心导航和知识阅读仍应正常工作。

### 搜索

Obsidian 原生搜索和 Omnisearch 用于补充链接导航。Omnisearch 主要用于快速定位页面并插入内部链接。

Omnisearch 的中文检索质量可能需要额外中文分词插件。该插件属于可选的 Vault 配置改进，不是知识库初始化的依赖。

### Excalidraw

Excalidraw 不属于第一版必需产物。它可以由用户在确实需要视觉表达时使用，但任何知识页面的核心含义都不能依赖 Excalidraw 文件。

## 入口页面

### 项目入口

`00 项目入口.md` 是人和 AI 的统一起点，但不是全部页面的完整目录。

它包含：

- 项目用途的简要说明。
- 系统最主要的运行路径。
- 三个架构页面的链接。
- 主要模块和流程视图的入口。
- 简短的任务导航，例如处理 RAG、Sandbox、文件预览、前端、配置或 API 问题时从哪里开始。
- 当前上下文页面的链接。

入口页不会链接每一篇笔记，避免重复动态视图并导致关系图形成无意义的中心节点。

### 当前上下文

`01 当前上下文.md` 是日常开发的交接页面，包含：

- 当前开发重点。
- 最近变化的领域。
- 当前风险和未解决问题。
- AI 处理当前工作时应优先阅读的页面。
- 最近一次维护使用的 Git 提交。

稳定结论应迁移到模块、流程、契约或决策页面。历史细节应迁移到变更记录。当前上下文始终保持短小和有效。

## 页面模板

### 模块页

模块页包含：

- 模块职责及明确边界。
- 相关源码目录和入口类。
- 关键组件及其作用。
- 上游、下游、参与流程和使用契约的链接。
- 真实运行机制。
- 配置和外部依赖。
- 约束、风险及开放问题。
- 代码和文档证据。

只有具有架构意义的类或机制才建立独立组件页。普通实现类保留在模块页的关键文件列表中。

### 流程页

流程页包含：

- 触发条件和最终结果。
- 按顺序排列的执行步骤。
- 参与模块的链接。
- 输入、输出和重要状态变化。
- 失败、中断、重试、降级和取消行为。
- 代码入口及证据。

### 组件页

组件页包含：

- 所属模块。
- 核心职责。
- 对外接口或运行时作用。
- 调用方和依赖。
- 生命周期或状态约束。
- 关键源码位置。

### 契约页

契约页包含：

- 生产方和使用方。
- 请求、响应、事件、配置或数据结构。
- 校验和兼容性约束。
- 失败行为。
- 权威 schema 或代码证据。

整理 AIO Sandbox HTTP 集成时，必须先阅读：

- `docs/sandbox-api/README.md`
- `docs/sandbox-api/integration-map.md`
- `docs/sandbox-api/openapi.json`

其中 `openapi.json` 是权威 REST 契约。

### 决策、风险和问题页

这些页面必须链接受影响的模块与流程。

- 决策页说明背景、最终选择、备选方案和后果。
- 风险页说明影响、证据、已知可能性和缓解方式。
- 问题页明确记录未知内容，以及可以如何验证。

## 证据与可信度

知识状态默认以页面为单位：

- `verified`：由代码、测试、配置、仓库内 schema 或权威项目文档直接确认。
- `inferred`：有价值但尚未完全证明的分析结论。
- `stale`：相关代码在最近复核后发生变化，或现有证据已经不匹配。

如果一个主要已验证的页面只包含少量推断，应在正文中显式标注该语句，而不是把整页降为 `inferred`。

证据引用使用仓库相对路径，并注明类名或方法名。必要时可以记录行号，但行号不是主要标识，因为它容易随代码修改而过期。

## 第一次完整整理

初始化会进行一次仓库级分析：

1. 阅读仓库规则、README、构建配置、项目文档和最近 Git 历史。
2. 盘点源码、测试、前端、配置、集成和基础设施。
3. 识别运行入口，并根据真实职责建立模块边界。
4. 梳理主要端到端流程和共享契约。
5. 创建完整目录和页面骨架。
6. 深入编写核心模块，为次要模块先提供简洁页面。
7. 添加属性、别名和有证据支持的内部链接。
8. 创建稳定知识视图和复核队列。
9. 填充项目入口和当前上下文。
10. 检查导航、链接、属性、关系图、证据和可信度标记。

完成标准：

- 从项目入口经过最多一个视图或架构页即可找到任意核心模块。
- 主要端到端流程都有独立页面。
- 核心模块的局部关系图反映有意义的依赖，而不是机械的入口链接。
- 重要结论能够追溯到仓库证据。
- 推断和过期内容可以通过复核视图找到。
- 已建立的知识不存在意外重复页面或失效链接。

## 持续维护

每次维护记录开始和结束的 Git 提交。

维护流程：

1. 读取上次记录后新增的提交和差异。
2. 把变化文件映射到受影响的模块、流程、契约和决策。
3. 阅读这些领域已有的知识页面。
4. 验证原有职责和关系是否仍然成立。
5. 更新正文、属性、别名和内部链接。
6. 将不确定或尚未复核的影响标记为 `inferred` 或 `stale`。
7. 更新当前上下文。
8. 为具有架构意义的变化添加简洁变更记录。
9. 检查受影响页面周围的局部关系，移除过时关系。
10. 记录新的 `source_commit`。

日常维护不重新扫描整个仓库。以下情况触发更大范围的重新评估：

- 大型重构。
- 模块移动或拆分。
- 重要依赖发生变化。
- 现有模块模型明显无法解释当前代码。

## 异常处理

- 如果目标 Vault 不存在或无法识别为 Obsidian Vault，在写入前停止。
- 如果生成内容会覆盖用户已经编辑的页面，必须先比较并有意识地合并，禁止直接覆盖。
- 如果仓库证据和现有知识冲突，应保留冲突并加入复核队列，直到得到确认。
- 如果关系无法验证，只能标记为推断，不能创建成已验证关系。
- 如果无法可靠划分某个模块，将其放入 `90 待整理` 并记录原因。
- 如果某个插件不可用，Markdown 页面和内部链接仍然必须可正常使用。

## 质量检查

初始化和维护流程需要检查：

- 必需属性是否存在，属性值是否合法。
- 页面名称是否唯一，别名是否有助于搜索。
- 内部链接是否有效。
- 稳定知识中是否存在意外孤立页面。
- 是否存在只为了美化关系图而添加的机械链接。
- 证据引用是否指向仓库内存在的路径。
- 从入口页面能否按阅读路径找到预期模块和流程。
- Bases 是否能正确显示预期分类。
- 当前上下文是否残留已经过期的历史细节。

在 Obsidian 中人工检查：

- 核心模块深度为一层或两层的局部关系图是否容易理解。
- 关系图过滤能否只显示当前项目和主要页面类型。
- 反向链接能否提供有价值的依赖上下文。
- 能否通过中文名、代码名和别名搜索到模块。

## 后续 Skill

第一次完整整理经过检查和修正后，再创建两个可复用 Skill：

- `project-kb-bootstrap`：为已有代码仓库建立 Obsidian 知识库。
- `project-kb-maintain`：根据 Git 变化更新已有知识库。

Skill 将固化经过验证的页面模板、属性规范、链接规则、证据标准、更新流程和质量检查。

具体项目的模块名称和架构结论属于分析结果，不会硬编码进通用 Skill。
