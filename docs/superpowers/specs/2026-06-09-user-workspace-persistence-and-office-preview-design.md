# 用户工作空间持久化与 Office 预览设计

## 状态

方案已于 2026-06-09 在对话中确认，等待用户审阅本文档。

## 目标

每个用户拥有一个跨 Agent、跨会话复用的独立工作空间。

第一阶段持久化并恢复：

- 用户的全部知识库原文件。
- 用户通过普通聊天或工作空间上传的文件。

同时支持在 AIO 沙箱中使用 LibreOffice 将 Office 文件转换为 PDF，
复用现有前端预览组件。

## 第一阶段范围

### 包含

- 持久化 `/home/gem/knowledge`。
- 持久化 `/home/gem/uploads`。
- 用户沙箱删除并重建后，从本地恢复以上两个目录。
- 本地存储和沙箱使用一致的相对目录与文件名。
- 复用现有重复文件处理：替换、保留两份、跳过。
- 保留知识库预览和普通文件预览的差异。
- Word、Excel、PowerPoint、OpenDocument 转换为 PDF 后预览。
- 继续支持现有 PDF、图片、文本、Markdown、CSV 和代码预览。
- 兼容当前已有的 `doc_{id}.{ext}` 知识库文件。

### 暂不包含

- 持久化 `/home/gem/workspace`。
- 持久化 `/home/gem/output`。
- 多服务器之间的分布式同步。
- 将沙箱内新生成的文件反向同步到本地。

## 数据来源原则

本地文件是持久化主副本，沙箱文件是可以重新生成的工作副本。

上传文件时：

1. 先保存本地文件。
2. 本地保存成功后，再同步到当前用户沙箱。
3. 沙箱同步失败时，不删除本地文件。
4. 下次创建沙箱时重新同步。

沙箱被删除或失效后，新沙箱从本地恢复文件。

## 统一目录结构

### 知识库文件

```text
本地：
uploads/users/{userId}/knowledge/{kbId}/{原始文件名}

沙箱：
/home/gem/knowledge/{kbId}/{原始文件名}
```

使用 `kbId` 隔离知识库，避免两个知识库中都存在 `合同.pdf` 时互相覆盖。

数据库继续保存用户看到的原始文件名，同时保存稳定的本地路径。
沙箱路径统一由 `kbId` 和清理后的原始文件名生成。

### 普通上传文件

```text
本地：
uploads/users/{userId}/uploads/{原始文件名}

沙箱：
/home/gem/uploads/{原始文件名}
```

普通上传从按 `sessionId` 隔离改为按 `userId` 隔离，因此用户切换 Agent
或会话后仍能看到相同文件。

## 重复文件处理

复用现有 `DuplicateHandler`：

1. 后端检测重复后返回 HTTP 409。
2. 前端让用户选择替换、保留两份或跳过。
3. 保留两份继续生成 `文件名 (1).pdf` 形式的新名称。
4. 最终确定的文件名必须同时用于本地文件、数据库元数据和沙箱文件。

不新增另一套重命名规则。

## 上传流程

### 知识库上传

```text
校验知识库归属
→ 检查当前知识库内的同名文件
→ 确定最终文件名
→ 创建文档数据库记录
→ 保存本地持久文件
→ 解析、切片、向量化并保存数据库切片
→ 当前用户沙箱可用时同步文件
→ 记录沙箱同步状态
```

知识库解析始终读取本地文件，不依赖沙箱是否可用。

### 普通文件上传

```text
根据 sessionId 查询 userId
→ 检查用户持久化上传目录中的同名文件
→ 复用现有重复文件处理
→ 保存本地持久文件
→ 当前用户沙箱可用时同步到 /home/gem/uploads
→ 返回统一的沙箱文件路径
```

替换文件时同时更新本地和沙箱副本。沙箱更新失败时，本地文件仍是主副本。

## 沙箱创建与恢复

新 AIO 沙箱就绪后依次执行：

1. 初始化 AIO 目录和工具脚本。
2. 将用户本地 `knowledge` 目录完整同步到 `/home/gem/knowledge`。
3. 将用户本地 `uploads` 目录完整同步到 `/home/gem/uploads`。
4. 同步当前会话启用的技能。
5. 记录同步结果。

文件同步必须使用 `byte[]` 或二进制流，不能继续使用
`Files.readString()`，否则 PDF、Office、图片等二进制文件会损坏。

单个文件同步失败时：

- 继续同步其他文件。
- 记录失败路径和汇总日志。
- 不删除本地文件。
- 后续预览或再次重建沙箱时允许重试。

## 旧知识库文件迁移

当前已有文件可能位于：

```text
uploads/knowledge/{userId}/doc_{docId}.{ext}
```

迁移规则：

1. 数据库中的旧 `storagePath` 仍可读取时，先把它作为原文件来源。
2. 首次同步或迁移时写入新路径：

```text
uploads/users/{userId}/knowledge/{kbId}/{原始文件名}
```

3. 新文件成功写入后才更新数据库 `storagePath`。
4. 新文件存在且大小正确后，才能删除旧文件。
5. 迁移中断后可以安全重试。

## 两种预览模式

前端继续共用现有 `FilePreviewer`，但必须保留两条不同的数据链路。

### 知识库预览

知识库预览继续显示左右双栏：

```text
左侧：
原文件，或者 Office 转换后的 PDF

右侧：
从 MySQL 的 knowledge_chunk 表读取的切片
```

接口保持逻辑分离：

```text
GET /api/rag/document/{docId}/file
GET /api/rag/document/{docId}/chunks
```

两个接口都必须校验文档属于当前登录用户。

Office 转换和切片加载互相独立：

- 左侧转换失败时，右侧数据库切片仍可显示。
- 右侧切片加载失败时，左侧文件仍可预览。

### 普通文件预览

普通工作空间文件只显示文件内容：

```text
GET /api/sessions/{sessionId}/files/preview?path=...
```

普通文件预览不查询、不显示知识库切片。

### 原文件类型与渲染类型

前端增加独立的渲染类型：

```text
fileType: docx
previewType: pdf
```

- `fileType` 用于标题、图标和下载原文件。
- `previewType` 决定使用 PDF、图片或文本预览器。

原生 PDF 的两个值都是 `pdf`。

## Office 转 PDF

### 支持格式

- Word：`doc`、`docx`、`odt`、`rtf`
- Excel：`xls`、`xlsx`、`ods`
- PowerPoint：`ppt`、`pptx`、`odp`

LibreOffice 在沙箱中以运行时 `gem` 用户执行：

```bash
timeout 120 soffice \
  --headless \
  --nologo \
  --nofirststartwizard \
  --norestore \
  -env:UserInstallation=file:///tmp/lo-profile-{唯一ID} \
  --convert-to pdf \
  --outdir "{输出目录}" \
  "{输入文件}"
```

每次转换使用独立的 LibreOffice 用户目录，避免并发转换时发生配置锁冲突。

所有输入和输出路径必须位于 `/home/gem` 下。不能把未经校验的用户路径直接
拼接到 Shell 命令中。

## 转换缓存

### 知识库缓存

```text
/home/gem/knowledge/{kbId}/.preview/{documentId}-{sourceVersion}.pdf
```

`sourceVersion` 根据源文件大小、修改时间或内容哈希生成。
替换源文件后缓存键变化，不会继续使用旧 PDF。

删除知识库文档时尽量同步清理预览缓存。

### 普通文件缓存

```text
/home/gem/temp/previews/{pathHash}-{sourceVersion}.pdf
```

路径哈希用于避免不同目录下的同名文件冲突。
普通预览缓存不需要持久化。

## 预览接口响应

Office 文件转换成功后返回：

- `Content-Type: application/pdf`
- `Content-Disposition: inline`
- 转换后的 PDF 内容
- 前端将 `previewType` 设为 `pdf`

下载操作仍返回原始 Office 文件，不返回转换后的 PDF。

## 失败处理

- 本地保存失败：上传失败。
- 沙箱同步失败：保留本地文件并记录可重试状态。
- 预览时沙箱文件缺失：先从本地主副本恢复一次，再返回错误。
- LibreOffice 转换超时：终止转换进程并返回明确错误。
- 不支持的文件类型：继续显示现有的下载提示。
- 旧文件迁移源不存在：标记同步失败，但不删除数据库切片。
- 知识库切片加载失败：只影响右侧切片区域。

## 安全要求

- 每次文件操作都要校验会话和文件属于当前用户。
- 知识库上传、预览、替换、删除、获取切片都校验所有权。
- 文件名只保留最后一个路径片段，阻止目录穿越。
- 普通预览路径限制在允许的 `/home/gem` 子目录内。
- LibreOffice 参数必须可靠转义或使用结构化命令调用。
- 浏览器接口不返回宿主机真实路径。

## 组件职责

### 用户文件存储服务

负责普通上传文件的用户级本地路径、二进制读写、重复检测、替换和删除。

### 知识库服务

负责知识库元数据、统一路径、解析、切片、向量、同步状态以及旧文件迁移。

### 文件同步服务

负责将本地目录按相对路径以二进制方式复制到当前用户沙箱，并返回成功和失败列表。

### 沙箱服务

负责在新沙箱创建成功后触发用户文件恢复，不在沙箱服务中重复实现文件命名规则。

### Office 预览服务

负责判断可转换格式、校验沙箱路径、管理缓存、调用 LibreOffice，并返回实际预览文件。

### FilePreviewer

负责区分知识库和普通文件模式，并按照 `previewType` 渲染。

## 验收标准

### 持久化恢复

1. 上传一个知识库 PDF 和一个普通图片。
2. 确认本地和沙箱相对路径一致。
3. 删除用户沙箱。
4. 创建新会话触发新沙箱。
5. 确认两个文件都按原字节恢复。

### 重复文件

知识库和普通上传都验证：

- 替换
- 保留两份
- 跳过

最终文件名在本地、数据库和沙箱中保持一致。

### 文件预览

- PDF、图片、文本、Markdown、CSV、代码维持原有行为。
- DOCX、XLSX、PPTX 能转换为 PDF 并预览。
- 知识库 Office 文件左侧显示 PDF，右侧继续显示数据库切片。
- 普通 Office 文件不请求知识库切片。
- 源文件未变化时复用转换缓存。
- 两个 Office 文件并发转换时互不冲突。

### 旧文件迁移

使用已有 `doc_{id}.{ext}` 文件验证：

- 可以恢复到新的统一沙箱路径。
- 新文件写入成功后才更新数据库路径。
- 迁移过程中知识库文档仍可用。
