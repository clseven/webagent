# 提示词架构重构设计

> 基于 `docs/lessons/` 中 s10（运行时组装）、s09（记忆注入）、s06（子代理隔离）的启发，
> 对现有 prompt 进行拆分 + 精简 + 动态化。

---

## 设计原则

1. **每条规则各管一件事，不重复** —— 同一个意思说 4 遍不会让 LLM 更重视，反而模糊重点
2. **原则 + 关键约束 > 逐步流程** —— 给模型判断框架，不替它做每一步
3. **行为指令不放 API 文档** —— 工具怎么用是工具定义的事，prompt 管什么时候用
4. **按需加载，不浪费 token** —— 基于 s10：section 加载取决于真实状态，不是硬编码

---

## ReactAgent System Prompt

### Section 加载策略

| Section        | 加载策略 | 判断依据                                              |
| -------------- | ---- | ------------------------------------------------- |
| identity       | 始终   | —                                                 |
| workspace      | 始终   | —                                                 |
| screenshot     | 按需   | `toolDefinitions` 含 browser_screenshot            |
| skill_system   | 按需   | `skillPrompt` 非空 或 `toolDefinitions` 含 skill_list |
| subagent       | 按需   | `toolDefinitions` 含 run_subagent                  |
| mcp_management | 按需   | `toolDefinitions` 含 mcp_add_or_update_server      |
| tools          | 始终   | 动态填入                                              |

### 拼接顺序

```
identity            ← 始终（最稳定，放最前以命中 prompt cache 前缀）
workspace 路径约定  ← 始终（静态）
screenshot          ← 按需（静态）
skill_system        ← 按需（静态）
subagent            ← 按需（静态）
mcp_management      ← 按需（静态）
tools               ← 始终（静态）
─── 动态边界 ───
workspace 目录树    ← 始终（每会话变化）
[技能元数据]        ← 按需（每会话变化，有 skillPrompt 时）
[任务策略]          ← 按需（每轮可能变化，有 plan 时）
```

静态段在前以命中 prompt cache 前缀；会话级易变段（目录树、技能元数据、任务策略）
集中在尾部，避免破坏前缀缓存。LLM 对首尾注意力都高，把当前任务上下文放尾部
既不丢注意力，又不毁缓存。

参考 CC 的 `SYSTEM_PROMPT_DYNAMIC_BOUNDARY`：boundary 之前的静态段合并为
global cache block，之后的动态段不参与 global cache。我们这里用空行 + 动态边界
注释模拟这种分隔，实际 cache 控制由调用层（Anthropic SDK 的 `cache_control`）负责。

### identity（始终加载）

```
你是沙箱环境中的执行者。遵循观察-行动-判断循环：
先看当前状态，再做一步行动，再根据反馈更新判断。

三条规则：
1. 事实必须来自工具，不用语言替代观察。下一步依赖上一步结果时串行调用，互不依赖时可并行。
2. 工具成功 ≠ 任务完成。回到成功信号找直接证据。
3. 计划与事实冲突时，改计划；用户否定结果时，改假设。

沙箱已由系统按会话准备，通过提供的工具访问，不假设宿主机状态。
回复时区分已完成、未完成和无法确认，只陈述有证据支持的结论。
```

**vs 原版（18 行 → 8 行）**：

- 原版把"用工具取事实"说了 4 遍（第 111、116、120-121、127 行），现在只说 1 遍
- 每条规则各管一件事：规则 1 管工具使用，规则 2 管完成判断，规则 3 管纠偏
- "观察-行动-判断循环" 一句话替代了原版 6 行的过程描述
- 规则 1 放开并行：Claude 4 系列原生支持并行 tool_use，硬约束"每次只调一个"
  会损失浏览器批量操作和多文件读取的效率；改为按依赖关系决策，更贴近"观察-行动-判断"

### screenshot（按需加载）

```
当用户需要亲自看到页面视觉内容（二维码、验证码、图形结果）时，
使用 browser_screenshot 将画面呈现给用户。页面状态可用 DOM 文本工具
（如 browser_get_html、browser_get_text）确认时，优先用文本工具而不是截图。
截图是观察和交付方式，本身不证明业务目标已达成。
```

**判断依据**：`toolDefinitions` 含 `browser_screenshot`。没有浏览器工具时不加载。

**vs 原版**：原版将截图说明混在 browser 工具的 description 里，模型在工具描述中
读到的偏向"怎么调"，缺少"什么时候不该调"的反向约束，容易把截图当默认观察手段。
独立成段后，加上"DOM 文本可确认时优先文本"的硬约束，避免无谓的图像 token 开销。

- 删掉了调用示例（`skill_activate(skill_id="brainstorming")`），模型从工具定义就知道怎么调

### mcp_management（按需加载）

```
## MCP 动态工具管理

目标环境固定为当前 WebAgent，不是 VS Code / Claude Desktop / Cursor 等外部客户端。

安装流程：搜索官方文档 → 确认 transport 和 endpoint → 展示方案等用户确认
→ 安装 → 用 mcp_list_servers 验证连接和工具列表。验证失败不得宣称成功。

关键约束：
- URL 必须来自官方配置的精确 endpoint，不猜测路径
- stdio MCP 在沙箱内运行（shell 命令和参数需完整填写，不要省略包名或留空参数）
- 历史中已展示的方案，用户说"确认"直接执行，不重复搜索
- MCP 子系统未启用 → 提示用户在配置中开启并重启 WebAgent 服务，不自行查找或修改配置文件
- 连接失败 → 根据错误码说明原因，修正时更新原 Server ID 不另建
- 新增的 MCP server 工具在下一条用户消息才会注册到工具列表，本轮不可调用
```

**vs 原版（10 条 ~600 token → 原则 + 6 条约束 ~200 token）**：

- 原版 10 条规则是踩坑后逐条加的补丁式写法，现在重构为：1 句目标 + 1 条安装流程 + 6 条不可省略约束
- 删掉了"安装前向用户展示来源、连接方式、可获得的能力和限制，等待用户明确确认"——安装流程里已经有"展示方案等用户确认"
- 删掉了"curl 或浏览器能访问 URL 只能证明网络可达，不能替代 MCP Client 验证"——安装流程最后一句已经覆盖
- 删掉了错误码的逐条解释（AUTH_REQUIRED、PROTOCOL_ERROR 等）——"根据错误码说明原因"交给模型自己判断

## PlanAgent System Prompt（精简）

### 原版分析

原版 PLANNER_SYSTEM_PROMPT 的核心问题：

- 和 ReactAgent 犯了同样的重复病——"不执行工具，也不替环境发言"说了 2 遍
- MCP 段和 ReactAgent 重复了大段内容
- 末尾的"只输出下面的任务模型"是好的，但前面太多铺垫冲淡了

### 重构版

```
你是任务的策略层。你的职责是帮助执行者建立一个清晰、可修正的任务模型，
不是预编排动作序列。

从用户请求和对话历史中提炼四个部分。你没有工具，不替环境发言，
没有经过工具观察的状态都不能描述成事实。

当用户纠正或否定先前结果时，重新检查假设，不自动重复之前的方案。

## 执行阶段能力

- 子代理：支持并行。当请求包含互不依赖的子任务或涉及耗时操作时，初始策略应建议并行推进。
- 技能：位于沙箱 /home/gem/skills/，执行阶段按需读取。未启用的 skill 需在前端启用。

%s

可用能力帮助你理解执行边界，但不要求在计划中逐项点名。
只输出下面的任务模型，不添加寒暄、执行叙述或额外段落。

### 目标状态
[用户真正希望实现的结果]

### 当前判断
[已知事实、关键假设和需要从环境确认的内容]

### 成功信号
[哪些可观察现象能够证明目标已经实现]

### 初始策略
[建议从哪里开始，以及为什么；保持简洁并允许执行者调整]
```

## 待确认问题（落地代码前需要回答）

以下问题不影响 prompt 文本本身，但会影响实现质量和上线后的可观测性。
本文档定稿前应该逐条对齐结论。

### 1. identity 第 1 条的并行工具调用边界

**问题**：原稿写"每次只调一个工具"是一刀切。Claude 4 系列原生支持并行 tool_use，
硬约束会损失浏览器批量操作、多文件读取的效率。本次已改为"下一步依赖上一步结果时
串行调用，互不依赖时可并行"。

**待确认**：

- 业务层是否真的禁止并行 tool_use？（比如沙箱 API 是否支持并发请求、是否会触发限流）
- 如果禁止，原因是什么？写"按需要并行，但每步都基于工具结果决策"是否更贴近原意？
- 如果允许，ReactAgent 的工具执行循环是否真的能处理一次 LLM 响应包含多个 tool_use block？

### 2. screenshot 段的来源说明

**问题**：原稿 screenshot 段没有"vs 原版"对比，看不出是新增段还是从某处精简而来。
本次已补上对比说明："原版混在 browser 工具的 description 里，独立成段是为了加上
'DOM 文本可确认时优先文本'的反向约束"。

**待确认**：

- 这个对比描述是否准确？需要核对原版 browser 工具定义里到底有没有截图使用说明
- 如果原版根本没说，那这一段是纯新增，文档措辞需要再调（去掉"vs 原版"，改成"新增理由"）

### 3. MCP 段"客户端 / 后端"指代不清

**问题**：原稿"客户端未启用 → 告知用户需重启后端"，对 LLM 来说"客户端"和"后端"
都是含糊词，容易理解错（比如把"客户端"理解成浏览器、把"后端"理解成沙箱）。
本次已改为：

- "MCP 子系统未启用 → 提示用户在配置中开启并重启 WebAgent 服务"
- "新增的 MCP server 工具在下一条用户消息才会注册到工具列表，本轮不可调用"

**待确认**：

- "WebAgent 服务"这个名词在用户视角是否成立？用户配置文件里到底叫什么名字？
- "MCP 子系统未启用"的具体错误信号是什么？模型能从哪个错误码 / 工具返回值识别这种情况？
- 是否需要把"下一条用户消息才注册"改成更精确的描述（比如"下一轮 agent loop"）？

### 4. 缺少可观测设计

**问题**：s10 教学版给了 `[assembled] sections: ...` 和 `[cache hit]` 的运行时打印，
让人能验证"按需加载"是否真的按需。本文档目前只在"实现要点"列了判断条件代码，
没有等价的运行时日志方案。上线后很难发现"按需"退化成"全加载"的 bug
（比如某个判断条件写错恒为 true，prompt 体积悄悄翻倍但没人察觉）。
